## PR Issue #6 - 커피 메뉴 목록 조회 API

### 작업 내용

커피 메뉴 목록 조회 API를 구현하고, 구현 후 Human 검토와 ChatGPT 1차 리뷰를 진행했다.

초기 구현에서는 Controller가 `Map<String, List<MenuResponse>>` 형태로 응답을 구성하고 있었고, 테스트 코드 역시 준비·실행·검증 흐름이 명확하게 구분되지 않았다.

Human 검토를 통해 응답 타입 안정성과 테스트 가독성을 개선했으며, 이후 ChatGPT 리뷰에서 Issue 범위 관리, 환경변수 설정, 비밀번호 하드코딩 문제를 추가로 점검했다.

## PR Issue #7 - 포인트 충전 API

### 작업 내용

포인트 충전 API의 예외 처리 구조를 검토하고 리팩토링했다.

기존에는 `PointChargeController` 내부에 여러 `@ExceptionHandler`가 작성되어 있어 Controller가 정상 요청 처리와 예외 응답 처리를 함께 담당하고 있었다.

1차 리뷰에서는 Controller의 예외 처리 로직을 `GlobalExceptionHandler`로 이동하는 수준으로 범위를 제한했다.

이후 개별 비즈니스 예외 클래스가 기능 추가에 따라 계속 증가할 수 있다는 점을 재검토했고, 비즈니스 예외는 `BusinessException` 하나로 통합하고 에러 정보는 `ErrorCode enum`에서 관리하는 구조를 적용하기로 결정했다.

단, 스프링 프레임워크가 직접 발생시키는 검증 및 JSON 파싱 예외는 `BusinessException`으로 변환하지 않고 `GlobalExceptionHandler`에서 별도로 처리한다.

### ChatGPT 리뷰

| 항목                | 리뷰 내용                                                                                                                     | 반영 여부 | 사유                                                                              |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------- | ----- | ------------------------------------------------------------------------------- |
| 예외 처리 책임 분리       | Controller 내부의 `@ExceptionHandler`를 `@RestControllerAdvice` 기반 `GlobalExceptionHandler`로 이동한다.                            | 반영    | Controller의 책임을 요청 처리와 서비스 호출로 제한하기 위함                                          |
| 공통 응답 형식 유지       | 예외 처리 위치와 내부 구조를 변경하더라도 기존 `PointErrorResponse` 형식은 유지한다.                                                                 | 반영    | API 응답 계약 변경을 방지하기 위함                                                           |
| HTTP 상태 코드 유지     | 기존 예외별 HTTP 상태 코드는 API 명세와 현재 구현을 기준으로 유지한다.                                                                              | 반영    | 클라이언트 호환성과 명세 일관성을 유지하기 위함                                                      |

| ErrorCode enum 도입 | HTTP 상태, 에러 코드, 기본 메시지를 `ErrorCode enum`에서 관리한다.                                                                          | 반영    | 예외 응답 정보를 한곳에서 관리하여 중복과 불일치를 줄이기 위함                                             |

| AI 제안과 Human 결정 혼재 | 최종 본문만 보면 어떤 내용이 AI 제안이고 Human 결정인지 구분하기 어려웠다. | 반영 | `AI 최초 제안 → Human 의견 → 최종 합의 → 근거` 표를 Issue에 추가 |

* AI가 도움이 된 부분

  * 현재 코드와 문서를 근거로 Human이 놓치기 쉬운 기간 경계, 동률, 0건 메뉴, LEFT JOIN 조건 위치를 제시했다.
  * 다른 계획 Issue에도 동일한 학습·검증 구조를 빠르게 확장했다.

* AI가 위험했던 부분

  * Human의 `다음 단계 진행`을 승인으로 잘못 해석해 Human 판단과 READY 전환을 대신했다.
  * 상세한 AI 초안이 최종 정답처럼 보일 수 있으므로 제안·확정 상태를 명시해야 한다.

* Human이 보완한 부분

  * 동률 시 `menuId ASC` 2차 정렬을 확정했다.
  * 주문이 0건인 메뉴도 LEFT JOIN으로 순위 후보에 포함해야 한다는 정책을 제시했다.
  * 모든 주문이 0건이어도 빈 목록이 아니라 0건 메뉴를 정렬해 TOP 3를 반환하도록 방향을 수정했다.

* 다음 Issue에서 개선할 점

  * AI는 빈 양식이 아니라 선택 가능한 상세 초안을 먼저 제공한다.
  * Human 작성 영역은 AI가 수정하지 않고 최초 이해 기록으로 보존한다.
  * AI 재검증은 한 번만 공식 기록한다.
  * 협의 후 Issue를 최종 구현 계약으로 한 번 정리한다.
  * Human의 명시적 승인 전에는 READY 전환과 Codex 구현 지시를 하지 않는다.
