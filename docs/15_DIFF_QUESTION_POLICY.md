# 실제 Diff 기반 질문 난이도 정책

## 1. 목적

이 문서는 Codex가 Pull Request의 최신 실제 Diff를 기준으로 작성하는 이해도 질문의 난이도와 범위를 통제한다.

질문의 목적은 구현자를 시험하거나 세부 구현을 암기시키는 것이 아니다. 작성자가 다음 내용을 자신의 말로 설명할 수 있는지 확인하는 것이다.

1. 이 변경이 해결하는 문제
2. 핵심 코드와 데이터 흐름
3. 적용한 기술의 역할
4. 테스트 결과가 의미하는 것
5. 현재 구현의 한계와 후속 개선 방향

질문은 신입 Java/Spring 백엔드 취업 준비생의 기술면접 수준을 유지한다. 위험도가 높아도 질문 자체의 개념 난이도를 높이지 않는다. 위험도는 질문 수와 확인할 실패 시나리오의 범위만 늘린다.

## 2. 적용 우선순위

실제 Diff 기반 질문을 생성할 때 다음 순서로 기준을 적용한다.

1. 해당 Issue의 작업 목적과 최종 계약
2. 실제 변경된 운영 코드와 핵심 테스트 결과
3. 이 문서의 난이도·구성 기준
4. `.github/PULL_REQUEST_TEMPLATE.md`의 질문 개수와 형식

질문 후보가 실제 Diff에 존재하더라도 Issue의 핵심 학습 목표와 관련이 약하면 제외한다.

예를 들어 동시성 정합성 Issue에서 테스트 데이터 삭제 순서, 임시 헬퍼 메서드 구현, Gradle DSL 세부 문법은 핵심 질문으로 사용하지 않는다.

## 3. 질문 개수

- LOW: 3개
- MEDIUM: 5개
- HIGH: 5~8개

질문 개수를 채우기 위해 주변 구현 세부사항을 억지로 질문하지 않는다. 적절한 핵심 질문이 부족하면 비슷한 질문을 늘리지 말고 변경 목적, 흐름, 결과 해석, 한계 중 아직 묻지 않은 관점을 사용한다.

## 4. 난이도 기준

모든 문항은 다음 조건을 충족해야 한다.

- 신입 개발자가 2~4문장으로 답할 수 있다.
- 한 문항에는 하나의 핵심 개념만 묻는다.
- 메서드명이나 설정 문법 자체가 아니라 그 코드의 목적과 동작을 묻는다.
- 해당 Issue를 구현하면서 실제로 사용하거나 관찰한 개념을 묻는다.
- 어려운 용어를 단독으로 묻지 않고 실제 흐름이나 테스트 결과를 함께 제시한다.
- 답변을 위해 프레임워크 내부 구현이나 운영 경험을 전제로 하지 않는다.
- 코드 한 줄의 이유보다 전체 기능의 문제·흐름·결과를 우선한다.

다음 질문은 신입 수준을 초과할 가능성이 높으므로 원칙적으로 피한다.

- 특정 동시성 도구를 둘로 나눈 세부 설계와 제거 시 발생하는 모든 오해를 동시에 설명하게 하는 질문
- Gradle Test task, DSL, 플러그인 내부 동작의 장단점을 상세 비교하는 질문
- Foreign Key 정리 순서, Fixture 삭제 순서처럼 Issue 핵심과 무관한 테스트 보조 코드 질문
- Spring, Hibernate, Tomcat, HikariCP 내부 구현을 암기해야 답할 수 있는 질문
- 아직 구현하지 않은 기술과 현재 구현을 심층 비교하는 질문
- 운영 장애, 대규모 트래픽, 튜닝 수치를 경험하지 않은 신입에게 단정하게 하는 질문
- 한 문항에서 원인, 대안, 장단점, 운영 위험을 모두 답하게 하는 복합 질문

## 5. 질문 구성 비율

전체 질문은 다음 비율을 우선한다.

- 변경 목적·핵심 흐름·결과 해석: 약 60%
- 기술 선택 이유와 직접 관련된 대안: 약 30%
- 코드·테스트 세부 구현: 10% 이하

LOW와 MEDIUM에서는 코드 세부 구현 질문을 최대 1개만 허용한다.

HIGH에서도 세부 질문을 늘리는 대신 권한, Rollback, 정합성, 메시지 유실처럼 실제 위험 시나리오를 추가한다.

## 6. 우선 질문 영역

아래에서 실제 변경과 직접 관련된 영역을 선택한다.

1. **문제 정의**
   - 이 변경 전 어떤 문제가 있었는가?
   - 왜 이 문제를 해결하거나 재현해야 했는가?

2. **핵심 흐름**
   - 요청이 들어온 뒤 어떤 계층과 데이터를 거치는가?
   - 변경 전후 흐름은 무엇이 달라졌는가?

3. **도메인 규칙과 정합성**
   - 어떤 값들이 서로 일치해야 하는가?
   - 테스트에서 관찰한 값이 왜 정상 또는 비정상인가?

4. **기술의 역할**
   - `@Transactional`, DB Lock, Cache, Event 같은 기술이 이 문제에서 어떤 역할을 하는가?
   - 기술 이름의 정의보다 현재 코드에서 맡은 책임을 묻는다.

5. **테스트 결과 해석**
   - 테스트가 무엇을 증명했는가?
   - 어떤 환경과 조건에서 실행했는가?
   - 무엇은 아직 검증하지 않았는가?

6. **후속 개선**
   - 다음 Issue에서 무엇이 어떻게 달라져야 하는가?
   - 동일한 테스트에서 어떤 값으로 회복되어야 하는가?

## 7. 질문 생성 절차

Codex는 질문을 PR에 작성하기 전에 다음 절차를 수행한다.

1. Issue의 작업 목적을 한 문장으로 정리한다.
2. 최신 Diff에서 핵심 운영 코드, 핵심 테스트, 핵심 결과를 구분한다.
3. 질문 후보를 문제·흐름·결과·기술·한계 영역에서 만든다.
4. 각 질문이 Issue 핵심 목적과 직접 연결되는지 확인한다.
5. 각 질문이 2~4문장으로 답할 수 있는지 확인한다.
6. 복합 질문이면 두 질문으로 나누거나 핵심 하나만 남긴다.
7. 주변 테스트 코드나 빌드 도구 세부사항 질문이 전체의 10%를 넘지 않는지 확인한다.
8. 난이도가 높은 질문은 쉬운 흐름·결과 중심 질문으로 교체한다.
9. 최종 질문만 PR 템플릿에 기록하고 Human 답변은 비워 둔다.

## 8. 생성 후 자체 검수 체크리스트

모든 항목을 만족하지 못하면 질문을 교체한다.

- [ ] 모든 질문이 Issue 작업 목적과 직접 관련된다.
- [ ] 변경 목적·핵심 흐름·테스트 결과 질문이 전체의 절반 이상이다.
- [ ] 한 질문에 핵심 개념이 하나만 있다.
- [ ] 신입 개발자가 2~4문장으로 설명할 수 있다.
- [ ] 프레임워크 내부 구현 암기를 요구하지 않는다.
- [ ] Issue와 무관한 Fixture 정리, 삭제 순서, Gradle DSL을 묻지 않는다.
- [ ] LOW·MEDIUM의 세부 구현 질문은 최대 1개다.
- [ ] 위험도가 높다는 이유만으로 질문 난이도를 높이지 않았다.
- [ ] 질문을 만들기 위해 아직 학습하지 않은 주변 기술을 끌어오지 않았다.

## 9. 좋은 질문과 나쁜 질문

### 동시성 정합성

나쁜 예:

```text
runConcurrently에서 ready와 start CountDownLatch를 분리한 이유와 하나가 없을 때 발생하는 모든 오해를 설명해 주세요.
```

좋은 예:

```text
동시 주문 테스트에서 여러 요청의 시작 시점을 맞춘 이유는 무엇인가요?
```

나쁜 예:

```text
deleteFixture가 outbox_events를 orders보다 먼저 삭제하는 이유와 잘못된 정리 순서의 격리 문제를 설명해 주세요.
```

좋은 예:

```text
주문 10건과 USE 이력 10건이 저장됐지만 잔액이 7,000P인 결과가 왜 Lost Update를 의미하나요?
```

### 트랜잭션

나쁜 예:

```text
Spring TransactionInterceptor와 EntityManager flush 순서를 설명해 주세요.
```

좋은 예:

```text
각 주문에 `@Transactional`이 적용돼 있는데도 여러 주문이 동시에 실행되면 정합성이 깨질 수 있는 이유는 무엇인가요?
```

### 테스트 분리

나쁜 예:

```text
Gradle의 Test task와 useJUnitPlatform includeTags/excludeTags 처리 원리를 설명해 주세요.
```

좋은 예:

```text
실제 MySQL 동시성 테스트를 일반 단위 테스트와 분리한 이유는 무엇인가요?
```

## 10. AI 역할 경계

- Codex가 실제 Diff 기반 공식 질문 세트를 한 번만 작성한다.
- ChatGPT는 별도의 경쟁 질문 세트를 추가하지 않는다.
- ChatGPT와 리뷰어는 기존 질문의 Issue 관련성, 난이도, 중대한 누락만 검수한다.
- 부적절한 질문은 Human이 답변하기 전에 교체한다.
- Human이 `모르겠다`고 답하면 AI는 실제 코드 흐름을 설명한다.
- AI는 모범 답안 재작성이나 정답 문장 복사를 요구하지 않는다.
- 단순 용어 부족, 표현 미숙, 세부 구현 기억 부족은 중대한 오해로 보지 않는다.
- 금액·포인트·재고 정합성, 권한, Rollback, 동시성 보호 대상처럼 사고로 이어지는 오해만 Human 리뷰어가 차단 여부를 판단한다.

## 11. 최종 원칙

질문은 구현자를 곤란하게 만드는 장치가 아니다.

```text
실제 Diff를 읽게 한다
→ 변경 목적과 핵심 흐름을 자기 언어로 설명하게 한다
→ 중요한 오해만 발견한다
→ 부족한 개념은 AI가 설명한다
→ Human이 최종 판단한다
```

세부 구현 암기보다 문제 정의, 데이터 흐름, 정합성, 테스트 결과 해석을 우선한다.
