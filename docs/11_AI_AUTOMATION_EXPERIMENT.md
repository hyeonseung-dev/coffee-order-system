# AI Automation Experiment

## 1. 실험 배경

현재 개발은 Human이 요구사항을 정의하고 ChatGPT와 Codex 사이를 직접 연결하는 Human-in-the-loop 방식이다. Human은 계획, 구현, 리뷰의 중간 단계마다 문맥을 전달하고 승인하며, 모든 작업에 유사한 수준의 검토를 적용한다.

이 방식은 학습과 안전성에는 유용하지만 자동화 수준이 낮아 작업이 늘어날수록 Human이 병목이 될 가능성이 있다. 따라서 이 개인 프로젝트를 AI 자동화 하네스의 실험 환경으로 사용해 자동화 효과와 위험을 함께 관찰한다.

## 2. 실험 목적

- 목표 입력부터 계획, 구현, 검증, 수정, PR 준비까지 연결되는 자동화 흐름을 구축한다.
- 작업 위험도에 따라 AI 자율성을 다르게 적용한다.
- 사람이 반드시 개입해야 하는 지점과 검토가 불필요했던 지점을 찾는다.
- 자동화 실패 사례, 원인, 재시도 결과를 기록한다.
- 관찰 결과를 바탕으로 최종 프로젝트에 적용할 AI 협업 기준을 도출한다.

> 이 실험의 목적은 사람을 완전히 제거하는 것이 아니라,
> 사람이 반복 작업의 병목이 되지 않으면서도
> 보안, 데이터 정합성, 설계 품질을 놓치지 않는 Human Gate 위치를 찾는 것이다.

## 3. 핵심 실험 질문

- 단순 CRUD는 사람의 중간 승인 없이 완료 가능한가?
- AI가 작성한 테스트만으로 구현을 신뢰할 수 있는가?
- 구현자와 검증자를 분리하면 오류 탐지율이 높아지는가?
- 테스트 통과 후에도 Human이 발견하는 문제가 있는가?
- 어떤 작업은 최종 diff 확인만으로 충분한가?
- 어떤 작업은 설계 단계부터 Human 승인이 필요한가?
- 몇 회까지 자동 재시도를 허용해야 하는가?
- 자동화 비용이 직접 검토 비용보다 커지는 시점은 언제인가?

## 4. 자동화 대상 범위

```text
목표 입력
→ 관련 문서 탐색
→ Issue 초안 또는 작업 계획 생성
→ 구현 계획
→ 코드 구현
→ 테스트 및 빌드
→ 독립 검증
→ 실패 시 수정
→ 재검증
→ PR 준비
→ 결과 보고
```

초기 단계에서 다음 작업은 자동화 범위 밖에 둔다.

- Merge
- 운영 배포
- 데이터 삭제
- DB Migration 실행
- 인증 정책 변경
- 비용이 발생하는 외부 시스템 작업

이는 영구 금지가 아니라, 실패 영향과 통제 방법을 확인하기 전까지 적용하는 **초기 실험 안전 경계**다.

## 5. 초기 자율성 단계

| 단계 | Human 역할 | AI 수행 범위 |
| --- | --- | --- |
| Level 1 — 제한 자동화 | Issue와 계획 승인, 코드 검토 | 구현과 테스트 |
| Level 2 — 개발 루프 자동화 | Issue 승인, 최종 결과 확인 | 계획, 구현, 테스트, 리뷰, 수정 |
| Level 3 — 작업 생성 포함 자동화 | 목표 제시, 위험 작업 승인 | Issue 초안, 계획, 구현, 검증, PR 준비 |
| Level 4 — 제한된 완전 자동화 | 사전 정의된 고위험 Gate 승인 | 작업 후보 탐색, Issue 생성 또는 선택, 구현, 검증, 수정, PR 준비 |

이번 개인 프로젝트의 우선 목표는 **Level 2**다. 후반부에는 관찰 결과상 위험이 통제된 일부 작업을 대상으로 **Level 3**을 실험한다. Level 4는 정의 대상이며 초기 달성 목표가 아니다.

## 6. 초기 위험도 가설

위험도는 Issue에 명시된 값을 Human이 승인한 최종 값으로 사용하며, 누락 시 `MEDIUM`이다. 위험도는 테스트·리뷰 강도, Rollback·정합성 검증, 최종 보고에만 사용하고 자동 BLOCKED나 재승인 근거로 사용하지 않는다.

| 위험도 | 작업 예시 | 검증 강도 |
| --- | --- | --- |
| LOW | 문서 수정, 오탈자, DTO, Mapper, 테스트 Fixture, 단순 조회, 반복 CRUD | 기본 검증과 최종 결과 확인 |
| MEDIUM | Service 비즈니스 로직, 예외 처리, Repository 쿼리, API 응답 계약, 캐시 조회 로직 | 관련 테스트와 독립 Reviewer 강화 |
| HIGH | 인증·인가, 금액·포인트, 트랜잭션, 동시성, 상태 전이, 캐시 정합성, DB Migration, 배포 | Rollback·정합성 검증과 독립 Reviewer 강화 |

## 7. Issue별 관찰 항목

각 자동화 실험은 다음 항목을 Attempt Log 또는 Verification Log에 기록한다.

| 구분 | 기록 항목 |
| --- | --- |
| 작업 식별 | Issue 번호와 작업명, 작업 위험도, 적용한 자율성 Level |
| Human 개입 | 개입 횟수, 개입 시점과 이유 |
| AI 실행 | AI 재시도 횟수, 재시도별 실패 원인과 변경 내용 |
| 검증 | 실행한 테스트와 검증, 독립 Reviewer 판정 |
| 최종 판정 | Human 최종 검토 결과, PASS·FAIL·BLOCKED |
| 누락과 효율 | AI가 놓친 문제, Human 검토가 불필요했던 부분, Human이 반드시 발견해야 했던 부분 |
| 다음 조정 | 다음 작업에서 적용할 자율성 수준과 Gate 변경 근거 |

## 8. 성공 기준

- [ ] 자동화된 개발 루프가 실제 Issue에서 동작한다.
- [ ] PASS, FAIL, BLOCKED 결과가 명확히 구분된다.
- [ ] 실패 이유와 재시도 결과가 기록된다.
- [ ] Human 개입 시점과 이유가 기록된다.
- [ ] 작업 유형별 적정 자율성 기준을 설명할 수 있다.
- [ ] 최종 프로젝트에 적용할 규칙과 제외할 규칙을 구분할 수 있다.

## 9. 실패 기준

- AI가 같은 오류를 반복하지만 원인이 기록되지 않는다.
- 테스트 통과만으로 잘못된 구현을 승인한다.
- Issue 범위 밖 변경이 반복된다.
- 자동화 관리 비용이 구현 비용보다 커진다.
- Human Gate가 형식적으로만 존재한다.
- 실패했는데도 PASS로 보고한다.
- 사람이 개입한 이유를 설명할 수 없다.

실패 기준에 해당한 실행은 숨기거나 성공으로 환산하지 않고 FAIL 또는 BLOCKED로 기록하며, 자동화 범위·재시도 한도·Human Gate 조정의 근거로 사용한다.

## 10. 최종 산출물

- AI 자동화 하네스 구조
- 역할별 에이전트 정의
- 실행 모드 또는 위험도별 워크플로우
- 검증 전략
- Attempt Log
- Verification Log
- Human Gate 기준
- 자동화 실패 사례
- 최종 프로젝트 적용안
- 면접에서 설명 가능한 AI 활용 사례

최종 평가는 자동화 단계의 높이가 아니라 반복 작업 감소, 오류 탐지, 통제 가능성, 기록 품질을 함께 기준으로 한다.
