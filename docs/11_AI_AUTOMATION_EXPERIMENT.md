# AI 자동화 실험: 멀티에이전트에서 단일 Codex 반자동화로

## 1. 실험 배경

기존 방식에서는 Human이 ChatGPT와 Codex 사이에서 요구사항, 구현 프롬프트, 결과, 수정 지시를 반복 전달했다.

```text
Human + ChatGPT 설계
→ Codex용 긴 프롬프트 작성
→ Codex 구현
→ Human이 결과 복사
→ ChatGPT 리뷰
→ Codex 수정
→ Commit·Push·PR을 단계별 지시
```

이 구조는 통제력은 높았지만 Human이 메시지 브로커가 되는 문제가 있었다.

- 같은 내용을 여러 번 전달해야 했다.
- 구현, 테스트, 수정, Commit, Push, PR 생성을 계속 따로 지시해야 했다.
- 실제 코드보다 AI 사이의 전달 형식을 관리하는 시간이 늘었다.

반복 비용을 줄이기 위해 저장소 안에 Agent, Skill, Hook, Log 규칙을 둔 자동화 하네스를 실험했다.

---

## 2. 처음 시도한 멀티에이전트 하네스

초기 구조는 역할별 에이전트를 분리하는 방식이었다.

```text
Issue
→ Preflight
→ Planner
→ Implementer
→ Verify
→ Reviewer
→ Log
→ PASS 또는 BLOCKED
```

실제 Issue에 적용하면서 다음 장점을 확인했다.

- Issue와 문서의 충돌을 구현 전에 발견했다.
- AI의 범위 밖 리팩터링을 통제했다.
- Rollback 통합 테스트와 실제 빌드를 실행했다.
- 구현과 독립된 관점의 리뷰를 받을 수 있었다.
- 로컬과 CI 환경의 차이를 발견했다.
- 실행한 테스트와 빌드 결과를 GitHub에 남겼다.

하지만 운영 비용도 컸다.

- Planner, Implementer, Reviewer가 같은 문서와 코드를 반복해서 읽었다.
- 역할 사이의 결과 전달과 상태 관리가 추가됐다.
- 위험도 재판정, 로그, PASS/BLOCKED 보고가 중복됐다.
- 환경 변수와 GitHub 도구 문제로 구현과 무관한 BLOCKED가 발생했다.
- 토큰, 실행 시간, Human 개입이 모두 증가했다.
- 하네스 관리가 실제 백엔드 개발만큼 커질 위험이 있었다.

자동화 단계가 많다고 반드시 더 좋은 개발 방식이 되는 것은 아니었다.

## 3. 현재 결론: 단일 Codex와 Human Gate

현재는 복잡한 Agent Teams와 Planner·Implementer·Reviewer 서브에이전트를 상시 사용하지 않는다.

하나의 Codex가 하나의 `READY` Issue를 처음부터 Draft PR까지 처리한다.

```text
Draft Issue
→ Human 착수 검토
→ ChatGPT 재검증
→ Human·AI 최종 합의
→ Human READY 승인

→ 단일 Codex
   Read
   → Preflight / 작업 브랜치 생성
   → Contract Traceability
   → Plan
   → Implement
   → Verify
   → Review Diff
   → Fix / Re-verify
   → Commit / Push / Draft PR

→ 위험도별 리뷰와 CI
→ 팀 이해·오해 점검
→ Human 직접 Merge
```

핵심은 에이전트를 여러 개 두는 것이 아니라 다음을 명확히 하는 것이다.

- AI가 언제 구현을 시작할 수 있는가
- 어떤 계약을 기준으로 구현하는가
- 무엇을 테스트하고 어떤 증거를 남기는가
- 어떤 문제는 AI가 해결하고 어떤 문제는 멈춰야 하는가
- 사람이 어느 지점에서 이해하고 결정해야 하는가

## 4. READY를 구현 승인으로 사용

`READY`는 단순히 Issue가 정리됐다는 뜻이 아니다.

다음이 이미 완료된 구현 계약이다.

- 담당자의 문제·범위·흐름 검토
- AI 재검증
- Human·AI 최종 합의
- 완료 조건과 테스트 기준 확정
- 포함·제외 범위 확정
- 위험도 결정
- 필요한 ADR 승인
- Human 구현 시작 승인

따라서 Codex는 READY Issue를 받은 뒤 짧은 계획을 보고하고 추가 승인을 기다리지 않는다.

```text
READY 확인
→ 계획과 영향 범위 보고
→ 구현 계속
```

문서와 코드가 충돌하거나 설계 재결정이 필요할 때만 `BLOCKED`로 전환한다.

이 변경으로 모든 단계마다 Human 승인을 반복하던 비용을 줄이면서 중요한 구현 전 판단은 유지했다.

---

## 5. Contract Traceability 추가

테스트 통과만으로 Issue 전체가 구현됐다고 판단할 수는 없다.

예를 들어 정상 응답 테스트가 통과해도 다음 조건은 빠질 수 있다.

- 시간대와 기간 경계
- 정렬과 동률 정책
- 트랜잭션과 Rollback
- 동시성 보호 대상
- Human이 확정한 구현 방식
- 제외 범위와 도입 금지 기술
- 직접 확인해야 하는 DB 또는 로그 상태

그래서 최종 계약을 원자 항목으로 나누고 실제 근거와 연결했다.

| ID | 최종 계약 | 실제 코드 증거 | 테스트·직접 검증 증거 | 상태 |
|---|---|---|---|---|
| C-01 | 구체적인 요구사항 또는 구현 제약 | `path#symbol` | 테스트 또는 직접 검증 | `PASS | HOLD | N/A` |

한 항목이라도 근거가 없으면 `HOLD`다.

```text
테스트 통과
≠ Issue 계약 전체 충족
```

`HOLD`가 남아 있으면 완료 보고, Commit, Draft PR 생성을 진행하지 않는다.

---

## 6. Troubleshooting Gate 추가

AI가 모든 오류마다 사람을 호출하면 자동화의 의미가 없다.

반대로 중요한 설계 문제까지 AI가 임의로 결정하면 위험하다.

### Codex가 자체 해결하는 문제

- 컴파일 오류
- import 누락
- 단순 테스트 데이터 오류
- 명확한 Mock 설정 오류
- 코드 스타일 오류
- 기존 패턴으로 해결 가능한 국소적 테스트 실패

### Codex가 중단하는 문제

- 요구사항을 바꿔야 테스트를 통과할 수 있음
- API·DB·트랜잭션·권한 정책 재결정 필요
- 동시성 정합성 미보장
- 데이터 손실 또는 중복 가능성
- 테스트 삭제·비활성화·약화 필요
- ADR과 구현 충돌
- 임시 우회와 근본 해결 중 선택 필요
- Issue 범위 밖 또는 보호 파일 변경 필요

중단할 때는 다음을 구분해 보고한다.

```text
확인된 사실
추정 원인
시도한 방법과 결과
영향 범위
선택지
권장 방향
Human이 결정할 사항
```

`BLOCKED`는 자동화 실패가 아니라 AI가 결정하면 안 되는 문제를 발견한 정상적인 결과다.

## 7. PR을 공식 인계 지점으로 변경

이전에는 Codex 결과를 Human이 다시 ChatGPT에 복사했다.

```text
Codex 구현
→ Human 결과 복사
→ ChatGPT 검토
```

현재는 Draft PR을 공식 인계 지점으로 사용한다.

```text
Codex 구현·검증
→ Commit·Push·Draft PR
→ Human이 PR 번호 전달
→ ChatGPT가 실제 Diff·Commit·Actions 검토
```

PR에는 단순 변경 파일 목록만 남기지 않는다.

- Human 최종 결정
- 구현 내용
- Contract Traceability
- 실행한 명령과 결과
- 테스트가 보장하는 것
- 테스트가 보장하지 않는 것
- 직접 검증과 미검증 항목
- 트러블슈팅과 알려진 제한
- Codex 자체 Diff 리뷰
- 실제 Diff 기반 이해도 질문

PR은 코드 제출서가 아니라 구현과 검증의 증거 문서가 된다.

---

## 8. Human 이해·오해 점검 추가

AI가 구현과 테스트를 모두 수행하면 개발자가 결과만 보고 넘어갈 위험이 있다.

이를 막기 위해 Codex가 실제 Diff와 테스트 결과를 기반으로 질문을 만든다.

- LOW: 3개
- MEDIUM: 5개
- HIGH: 5~8개

질문은 다음을 확인한다.

- 해결한 문제
- 적용 전과 적용 후 코드·데이터 흐름
- 핵심 도메인 규칙의 구현 위치
- 정상·실패·Rollback 시 데이터 상태
- 테스트가 보장하는 것과 보장하지 않는 것
- 선택한 대안과 트레이드오프
- 현재 구현의 한계

작성자와 리뷰어는 실제 Diff를 읽고 자신의 말로 답한다.

모르는 내용은 모른다고 작성할 수 있다.

AI는 사람의 이해도를 `PASS | HOLD`로 채점하지 않는다. 실제 코드와 답변을 대조해 다음만 제공한다.

- 코드와 일치하는 부분
- 보완 설명이 필요한 부분
- 미검증 사항과 잔여 위험
- 중대한 오해 후보

중대한 오해인지, Merge를 보류할지는 Human 리뷰어가 최종 판단한다.

처음에는 AI에게 하네스를 걸기 위해 규칙을 만들었지만, 결과적으로 사람도 코드를 이해하지 않고는 넘어갈 수 없게 됐다.

```text
AI는 승인된 계약 없이 구현할 수 없다.
Human도 이해하지 않은 코드를 Merge할 수 없다.
```

## 9. 위험도는 검증 강도를 조절

위험도는 에이전트 수를 늘리는 기준이 아니다.

### LOW

- Codex 자체 Diff 리뷰
- 관련 테스트와 CI
- 작성자 답변
- 비담당 리뷰어 1명 확인

### MEDIUM

- 정상·실패·경계 테스트
- Contract Traceability
- 작성자 답변
- 비담당 리뷰어 최소 1명 이해 요약
- 필요 시 ChatGPT 또는 독립 AI 리뷰
- Human 리뷰와 CI

### HIGH

- 구현 전 설계·트레이드오프·ADR 확인
- 강화된 권한·Rollback·동시성·정합성 테스트
- Contract Traceability
- 작성자 답변
- 비담당 리뷰어 최소 2명 이해 요약
- 독립 AI 리뷰 필수
- 강화된 Human 리뷰와 CI

HIGH에서 독립 AI 리뷰를 추가하더라도 상시 Reviewer 서브에이전트를 운영하는 것은 아니다.

위험한 PR에만 구현 문맥과 분리된 관점을 추가한다.

## 10. Merge는 Human 전용

Merge는 처음부터 자동화 대상이 아니었고 현재도 예외가 없다.

```text
ChatGPT Merge 금지
Codex Merge 금지
자동 Merge 금지
Human만 Merge 가능
```

AI의 PASS, 리뷰 완료, Merge 가능 의견은 참고 자료일 뿐이다.

Human은 다음을 직접 확인한다.

- Issue 완료 조건
- Contract Traceability 전체 PASS
- 최신 Commit 기준 CI
- 위험도별 리뷰
- 문서와 코드 정합성
- 범위 밖 변경
- 중대한 오해
- 미검증 항목과 잔여 위험

최종 선택과 결과 책임은 사람에게 남는다.

---

## 11. 최종 평가

처음에는 멀티에이전트 구조가 더 정교하고 안전할 것이라고 생각했다.

실제 적용 결과, 역할 분리는 일부 검증 품질을 높였지만 같은 문맥을 반복해서 읽고 관리하는 비용도 크게 늘렸다.

현재 결론은 다음과 같다.

```text
설계와 위험 판단은 Human
문서와 코드 교차 검증은 ChatGPT
승인된 구현과 반복 검증은 단일 Codex
최종 판단과 Merge는 Human
```

좋은 AI 워크플로우는 에이전트 수로 결정되지 않는다.

- 반복 전달을 줄이는가
- Issue 범위를 지키는가
- 테스트 결과를 과장하지 않는가
- 요구사항과 코드의 연결 근거가 남는가
- 중요한 문제에서는 AI가 멈추는가
- 담당자가 반영한 코드를 설명할 수 있는가
- 자동화 운영 비용이 실제 개발을 방해하지 않는가

현재 하네스는 모든 실수를 막는 완전 자동화 시스템이 아니다.

**사람이 이해하고 책임지는 백엔드 결과물을 만들기 위한 최소 안전장치다.**

🤖 **AI 활용**

- ChatGPT로 Issue, 문서, 코드의 충돌과 설계 누락을 검토했다.

- AI의 결과를 그대로 신뢰하지 않고 Contract Traceability, 실제 테스트 증거, Human 설명을 함께 확인했다.
