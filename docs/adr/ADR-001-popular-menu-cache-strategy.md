# ADR-001: 인기 메뉴 조회에 DB 집계 결과 캐시를 사용한다

## 상태

`Accepted`

## 작성일

2026-07-17

## 관련 Issue

- #12 인기 메뉴 조회 인덱스 적용 전후 성능 비교
- #13 Redis 캐시 적용 및 TTL·Hit/Miss 검증
- #45 K6 기반 인기 메뉴 API 부하 테스트 및 성능 비교

## 배경

최근 7일 인기 메뉴 TOP 3 조회는 주문 데이터를 기간 조건으로 집계하고, 메뉴별 주문 수를 정렬하는 읽기 작업이다.

반복 조회 시 MySQL에서 동일한 집계 쿼리를 계속 실행하면 DB 부하가 커질 수 있으므로 Redis 적용 방식을 결정해야 한다. 주요 대안은 다음 두 가지다.

1. MySQL 집계 결과를 Redis에 캐시하는 방식
2. 주문 시점마다 Redis Sorted Set(ZSet) 점수를 갱신해 랭킹 원본처럼 사용하는 방식

Redis ZSet은 점수 증가와 상위 순위 조회가 빠르고 원자적이다. 다만 MySQL 주문 저장과 Redis 점수 갱신은 하나의 트랜잭션으로 묶이지 않으므로, 정확한 랭킹을 유지하려면 이중 쓰기 실패와 재처리를 별도로 다뤄야 한다. 최근 7일 범위를 유지하려면 기간이 지난 주문 제거, 취소·환불 보정, 시간 경계, 장애 후 재구축 정책도 필요하다.

현재 요구사항은 실시간 순위 변동을 수초 이내에 제공하는 것이 아니라, 최근 7일 인기 메뉴 TOP 3 결과를 일정 시간 동안 빠르게 반복 조회하는 것이다.

## 결정해야 할 사항

최근 7일 인기 메뉴 TOP 3 조회에 Redis ZSet 실시간 집계를 사용할지, MySQL 집계 결과를 Cache-Aside 방식으로 캐시할지 결정한다.

## 고려한 대안

### 대안 1: MySQL 집계만 사용

- 설명: 요청마다 MySQL에서 최근 7일 주문을 집계하고 정렬한다.
- 장점:
  - MySQL `orders`만 정답 데이터로 사용하므로 정합성 구조가 단순하다.
  - 캐시 무효화와 Redis 장애 정책이 필요 없다.
- 단점:
  - 동일한 집계 쿼리가 반복 실행된다.
  - 데이터와 요청량이 증가하면 DB CPU, I/O, 정렬 비용이 커질 수 있다.
- 위험:
  - 인덱스만으로 충분하지 않은 트래픽에서도 DB 조회가 집중될 수 있다.

### 대안 2: MySQL 집계 결과를 Redis에 캐시

- 설명: Cache Miss 시 MySQL에서 집계하고, 완성된 TOP 3 응답을 Redis에 TTL과 함께 저장한다. Cache Hit 시 저장된 결과를 바로 반환한다.
- 장점:
  - 캐시 Hit 시 집계와 정렬을 다시 수행하지 않는다.
  - Redis는 파생 데이터이므로 장애나 유실 시 MySQL에서 재생성할 수 있다.
  - 주문 저장 경로에 Redis 쓰기를 추가하지 않아 주문 트랜잭션과 분리된다.
  - 구현과 장애 복구 기준이 단순하다.
- 단점:
  - Cache Miss 또는 TTL 만료 시 DB 집계가 필요하다.
  - TTL 동안 최신 주문 반영이 지연된다.
  - 동시 Cache Miss가 발생하면 동일 집계가 중복 실행될 수 있다.
- 위험:
  - TTL과 동시 미스 제어를 부적절하게 설정하면 DB 부하가 순간적으로 증가할 수 있다.

### 대안 3: Redis ZSet 실시간 랭킹

- 설명: 주문 상태 변경 시 메뉴별 점수를 `ZINCRBY` 등으로 갱신하고, `ZREVRANGE` 계열 명령으로 상위 메뉴를 조회한다.
- 장점:
  - 주문 시점에 점수를 미리 계산하므로 지속적인 랭킹 조회 처리량이 높다.
  - 상위 N개, 특정 메뉴 순위, 점수 조회 등 랭킹 기능 확장에 유리하다.
  - Redis 내부의 점수 증가와 순위 조회는 빠르고 원자적이다.
- 단점:
  - MySQL 주문과 Redis 점수의 이중 쓰기 정합성을 직접 보장해야 한다.
  - 최근 7일 윈도우 제거, 취소·환불 보정, 시간 경계 관리가 필요하다.
  - 장애 후 DB 기준 재구축과 중복 없는 이벤트 재처리 정책이 필요하다.
- 위험:
  - DB에는 주문이 있지만 Redis 반영이 누락되거나, DB 롤백·취소 후 Redis 점수만 남아 실제 순위와 달라질 수 있다.
  - 정확성을 확보하려면 Outbox, 재시도, 멱등성, 재구축·대사 작업 등 별도 실시간 집계 시스템 수준의 운영 비용이 발생한다.

## 성능 및 비용 비교

| 비교 항목 | DB 집계 결과 캐시 | Redis ZSet 실시간 랭킹 |
| --- | --- | --- |
| Cache Hit / 랭킹 조회 | 완성된 결과를 `GET`하므로 매우 빠름 | 상위 N개 조회가 매우 빠르고 데이터 규모가 커져도 일정한 편 |
| Cache Miss | MySQL 집계와 정렬 비용 발생 | 별도 집계 없이 바로 조회 가능 |
| 최신성 | TTL 범위만큼 지연 허용 | 이벤트가 정상 반영되면 실시간에 가까움 |
| 주문 쓰기 경로 | MySQL 주문 저장만 수행 | MySQL 저장과 Redis 점수 갱신 또는 이벤트 처리 필요 |
| 정합성 책임 | MySQL이 원본이며 캐시는 재생성 가능 | MySQL과 Redis의 반영 일치, 취소 보정, 중복 방지 필요 |
| 최근 7일 관리 | 조회 조건으로 계산 | 만료 대상 제거 또는 시간 버킷·재집계 구조 필요 |
| Redis 장애 | MySQL 조회로 Fallback 가능 | 랭킹 누락·복구·재구축 기준 필요 |
| 구현·운영 비용 | 낮음 | 높음 |
| 적합한 요구사항 | 일정 시간 지연을 허용하는 반복 TOP N 조회 | 수초 이내 갱신이 필요한 지속적 실시간 랭킹 |

Redis ZSet은 주문 시점에 점수를 미리 갱신하므로 지속적인 실시간 랭킹 조회에서는 더 높은 처리량과 일정한 조회 성능을 제공한다. 그러나 현재 요구사항은 최근 7일 인기 메뉴 결과를 일정 시간 캐싱하는 것으로 충분하며, 캐시 Hit 시 조회 성능 차이가 크지 않다. 따라서 실시간성을 위해 추가되는 정합성 및 운영 비용을 고려해 DB 집계 결과 캐시를 선택한다.

여기서 비용은 Redis 명령 자체의 실행 비용만 의미하지 않는다. 이중 쓰기 실패 처리, 이벤트 재시도와 멱등성, 윈도우 만료, 장애 복구, 데이터 대사, 테스트와 모니터링까지 포함한 구현·운영 비용을 의미한다.

## 결정

최근 7일 인기 메뉴 TOP 3 조회는 다음 구조를 사용한다.

```text
요청
  → Redis Cache 조회
      ├─ Hit: 캐시된 TOP 3 반환
      └─ Miss: MySQL에서 최근 7일 집계
                   → 결과를 Redis에 TTL과 함께 저장
                   → 응답 반환
```

- 패턴: Cache-Aside
- 정확한 원본: MySQL `orders`
- Redis 역할: 계산된 인기 메뉴 응답 캐시
- 주문 성공 시 Redis 점수를 증가시키지 않는다.
- 주문 성공 시 캐시를 즉시 Evict하지 않고 TTL 기반 갱신을 허용한다.
- Redis 조회 실패 시 MySQL 집계로 Fallback한다.
- Redis 데이터는 언제든 MySQL 기준으로 재생성할 수 있어야 한다.

## 결정 이유

1. 현재 요구사항은 실시간 랭킹보다 일정 시간 지연을 허용하는 TOP 3 반복 조회에 가깝다.
2. 캐시 Hit에서는 완성된 결과를 한 번 읽으므로 ZSet 상위 조회와 비교해 사용자 체감 성능 차이가 크지 않을 가능성이 높다.
3. ZSet을 정확하게 운영하려면 단순 자료구조 적용을 넘어 주문·취소 이벤트 기반 집계 시스템이 필요하다.
4. 프로젝트의 우선 학습 목표는 주문·포인트 트랜잭션, 동시성, DB 인덱스, Cache Miss·Hit와 장애 Fallback 검증이다.
5. 현재 규모에서는 ZSet 정합성 구축 비용보다 DB 집계 결과 캐시의 단순성, 복구 가능성, 검증 가능성이 더 가치가 크다.
6. 실제 성능 우위는 추측하지 않고 #45에서 인덱스 적용 MySQL, Cache Miss, Cache Hit를 동일 조건으로 측정한다.

## 영향

- 코드:
  - 인기 메뉴 조회 서비스와 별도의 Redis 캐시 컴포넌트를 사용한다.
  - 주문 생성·취소 코드에서 ZSet 점수를 직접 갱신하지 않는다.
- 데이터·DB:
  - MySQL `orders`가 인기 메뉴 결과의 유일한 정답 데이터다.
  - 인덱스와 집계 쿼리 실행계획은 별도로 검증한다.
- 트랜잭션·정합성:
  - 주문 트랜잭션과 Redis 캐시 저장을 강한 원자성으로 묶지 않는다.
  - 캐시 쓰기 실패는 주문 실패로 전파하지 않는다.
- 테스트:
  - Cache Miss, Hit, 빈 결과 Hit, TTL 만료, Redis 읽기·쓰기 실패와 MySQL Fallback을 검증한다.
- 성능:
  - Cache Hit는 반복 DB 집계를 제거한다.
  - Cache Miss 비용은 남으며 실제 개선 폭은 K6와 실행계획 결과로 판단한다.
- 운영:
  - Redis 장애 중에도 MySQL 원본 조회로 기능을 유지한다.
  - Redis 복구 후 자연스러운 Cache Miss를 통해 데이터를 다시 채운다.
- 문서:
  - Redis ZSet을 채택하지 않은 이유와 재검토 조건을 이 ADR에 보존한다.

## 검증 계획

- 테스트:
  - Cache Miss 시 Repository 조회 1회와 캐시 저장을 검증한다.
  - Cache Hit 시 Repository 조회가 발생하지 않는지 검증한다.
  - TTL 만료 후 다시 MySQL을 조회하는지 검증한다.
  - Redis 장애 시 MySQL 결과를 반환하는지 검증한다.
- 직접 검증:
  - Redis Key, TTL, 직렬화 결과를 확인한다.
  - MySQL 데이터와 캐시 응답이 정책상 허용한 시점에서 일치하는지 확인한다.
- 성능·동시성 검증:
  - 인덱스 적용 MySQL 조회, Cache Miss, Cache Hit를 동일 데이터와 부하 조건에서 비교한다.
  - 평균, p95, 처리량, 실패율을 함께 기록한다.
  - 동시 Cache Miss 시 중복 집계 여부를 관찰하고 필요할 때만 추가 제어를 검토한다.
- 성공 기준:
  - Cache Hit에서 DB 집계가 발생하지 않는다.
  - Redis 장애가 주문 기능과 인기 메뉴 조회 전체 장애로 전파되지 않는다.
  - 측정 결과와 실행 환경을 문서에 기록하며 개선 효과를 과장하지 않는다.

## 알려진 한계

- TTL 동안 신규 주문 또는 상태 변경이 인기 메뉴 결과에 즉시 반영되지 않는다.
- Cache Miss에서는 DB 집계 비용이 남는다.
- 단일 Redis 캐시만으로 다중 장애 상황의 고가용성이 보장되지는 않는다.
- 부하 테스트 전에는 운영 환경에서의 성능 우위를 확정할 수 없다.

## 재검토 조건

다음 조건 중 하나가 실제 요구사항이나 측정 결과로 확인되면 Redis ZSet 또는 별도 실시간 집계 구조를 다시 검토한다.

- 인기 메뉴 순위가 수초 이내에 반영되어야 한다.
- TOP 3 외에 임의의 TOP N, 특정 메뉴의 현재 순위, 순위 변동 조회가 핵심 기능이 된다.
- Cache Miss 집계 비용이 목표 응답시간 또는 DB 처리량 기준을 반복적으로 초과한다.
- 읽기 트래픽이 매우 커서 TTL 결과 캐시만으로 DB 부하를 제어하기 어렵다.
- Outbox, 메시지 브로커, 멱등 Consumer, 재구축·대사 체계를 운영할 필요성과 여력이 생긴다.

재검토 시에는 ZSet의 자료구조 성능만 비교하지 않고, 정확한 최근 7일 윈도우와 장애 복구까지 포함한 전체 시스템 비용을 비교한다.
