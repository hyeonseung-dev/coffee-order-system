# AI-assisted Development Workflow

## 1. 목적

이 프로젝트는 완전 자동화를 목표로 하지 않는다.

AI는 설계 문서와 백로그 초안 작성, 반복 구현과 검증 속도를 보조한다. Human은 문제 정의, 도메인 규칙, 설계와 트레이드오프, 테스트 기준, 트러블슈팅 의사결정, 최종 선택과 결과 책임을 맡는다.

핵심 원칙은 다음과 같다.

```text
AI가 초안을 만든다.
Human이 구현 전 다시 생각한다.
ChatGPT가 누락·충돌·대안을 검토한다.
Human이 요구사항·범위·설계를 최종 결정한다.
Codex는 승인된 READY Issue만 구현한다.
Issue는 작업 계약이다.
PR은 공식 인계 지점이다.
테스트와 검증 없는 완료는 완료가 아니다.
Merge는 Human만 수행한다.
```

## 2. 역할 분리

### Human

- 서비스 목적과 우선순위 결정
- 요구사항, 도메인 정책, 전체 설계 승인
- 실제 구현할 Issue 선정
- 구현 전 Issue를 자신의 언어로 재해석
- 문제, 핵심 규칙, 설계, 트레이드오프, 테스트·검증 계획 작성
- 위험도와 Human Gate 판단
- 중요한 기술 선택과 ADR 승인
- AI가 작성한 코드 이해와 검토
- 트러블슈팅 선택지의 최종 결정
- 최종 기능 확인과 Merge 직접 수행
- 결과 책임

### ChatGPT

- 요구사항, API, ERD, ADR 등 설계 문서 작성과 검토 지원
- 설계 문서를 근거로 Draft Issue와 일정 초안 작성
- Human 사전 검토의 누락·충돌·위험·대안 제시
- Human 결정을 반영한 최종 Issue와 Markdown 문서 정리
- ADR 초안 작성과 대안 비교 지원
- GitHub PR의 실제 diff, 커밋, Actions, 테스트 결과 검토
- 코드 리뷰와 수정 지시 구조화
- 구현 결과에 종속되는 문서 보완
- 중요한 충돌 발견 시 문서를 임의로 맞추지 않고 Human에게 보고

ChatGPT는 초안을 빠르게 만들고 검토를 돕지만, 중요한 요구사항·범위·설계를 대신 결정하지 않는다.

### Codex

- 승인된 READY Issue의 Java/Spring 코드 구현
- 테스트 코드 작성
- 로컬 실행, 컴파일, 디버깅, 테스트, 빌드
- 일반적인 오류의 자체 해결
- Commit, 작업 브랜치 Push, PR 생성
- 리뷰 수정 사항을 같은 PR에 반영
- 중요한 설계·정합성·보안·성능 문제 발견 시 Troubleshooting Gate 보고

### GitHub

- Issue로 요구사항과 완료 조건 관리
- ADR과 문서로 설계 근거 보존
- Branch와 Commit으로 구현 이력 관리
- PR로 Codex 결과를 ChatGPT와 Human에게 인계
- Actions로 테스트와 빌드 결과 제공
- 리뷰와 Human Merge 이력 보존

## 3. 전체 Workflow

### 0단계: 프로젝트 설계 기준 준비

Human과 ChatGPT가 구현 전에 프로젝트 전체 기준을 정리한다.

주요 문서:

- 프로젝트 목표와 요구사항
- 핵심 도메인 규칙
- API 명세
- ERD
- 트랜잭션과 데이터 정합성 정책
- 테스트 전략
- 중요한 설계 결정을 기록한 ADR
- AGENTS, Skill, Agent, Hook, Log, Issue·PR 템플릿

이 문서들은 AI에게 모든 판단을 맡기기 위한 것이 아니라 Human과 AI가 같은 기준으로 작업하기 위한 기준선이다.

### 1단계: Draft Backlog

ChatGPT가 설계 문서를 근거로 주요 기능의 Draft Issue와 일정 초안을 작성한다.

Draft Issue는 변경 가능한 백로그이며 구현 계약이 아니다.

최소 항목:

- 작업 목적
- 예상 범위
- 예상 완료 조건
- 선행 Issue
- 예상 위험도
- 미정 사항

Human은 다음을 검토해 백로그와 우선순위를 확정한다.

- 필수 기능이 우선인지
- Issue의 크기와 선후 관계가 적절한지
- 일정 안에 가능한지
- 도전 기술이 핵심 기능보다 앞서지 않는지
- 불필요하거나 과도한 범위가 없는지

### 2단계: Human Pre-Refinement

Human은 다음 구현 Issue를 선정한 뒤 AI Draft를 그대로 승인하지 않는다.

관련 설계 문서와 현재 코드를 확인하고 다음 내용을 자신의 언어로 작성한다.

- 해결하려는 문제
- 왜 지금 필요한지
- 핵심 도메인 규칙
- 예상한 설계와 구현 흐름
- 고려한 대안과 트레이드오프
- 완료 조건
- 제외 범위
- 필요한 테스트와 검증
- 예상 트러블슈팅
- 검토가 필요한 부분

확신하지 못하는 내용은 AI가 임의로 채우게 하지 않고 `검토 필요`로 남긴다.

LOW 위험도 작업은 필요한 항목만 간단히 작성한다. 트랜잭션, 동시성, 금액, 권한, 캐시, 이벤트처럼 중요한 작업은 실패 시나리오와 정합성까지 구체적으로 작성한다.

### 3단계: Collaborative Issue Refinement

ChatGPT는 Human 사전 검토와 설계 문서·현재 코드 사이의 정합성을 확인한다.

검토 항목:

- 요구사항과 핵심 규칙 누락
- API·ERD·ADR과의 충돌
- 정상 흐름과 실패 시나리오
- 트랜잭션과 데이터 정합성
- 동시성, 보안, 성능 위험
- 과도한 구현 범위
- 테스트가 실제 규칙을 증명하는지
- 테스트와 별개로 필요한 직접 검증
- 기술 선택의 대안과 트레이드오프

진행 방식:

```text
Human 사전 검토
→ ChatGPT가 질문·문제점·대안 제시
→ Human이 요구사항·범위·설계 결정
→ ChatGPT가 결정 결과를 Issue에 정리
```

ChatGPT의 제안은 자동 확정안이 아니다. 최종 판단은 Human이 내린다.

### 4단계: ADR 판단

모든 Issue에 ADR을 작성하지 않는다.

다음 조건 중 하나에 해당할 때만 작성한다.

- 둘 이상의 합리적인 대안이 존재함
- 선택 결과가 여러 기능이나 계층에 영향을 줌
- 변경 비용이 크거나 되돌리기 어려움
- 트랜잭션, 동시성, 보안, 캐시, 메시징, 배포 관련 결정임
- 면접이나 회고에서 선택 근거를 남길 가치가 있음

ChatGPT가 대안과 장단점을 정리하고 Human이 선택한다. 선택한 결과를 ADR로 기록한다.

ADR은 원칙적으로 Collaborative Refinement와 READY 승인 사이에 작성한다. 구현 중 새로운 중요한 선택이 발견되면 Codex가 멈추고 ADR과 Issue를 갱신한 뒤 Human 재승인을 받는다.

### 5단계: Ready Check와 Human 승인

다음 조건을 통과해야 Issue가 `READY`가 된다.

- 담당 Human이 Issue를 자신의 언어로 해석함
- Human이 문제, 규칙, 설계, 트레이드오프, 테스트·검증을 검토함
- ChatGPT 검토 후 Human이 요구사항·범위·설계를 최종 결정함
- 목적과 완료 조건이 명확함
- 완료 조건을 테스트하거나 검증할 수 있음
- 제외 범위와 수정 경계가 명확함
- 선행 Issue가 완료됨
- API·ERD·설정 영향이 확인됨
- 위험도와 Human Gate가 결정됨
- 테스트·검증·트러블슈팅 계획이 있음
- 필요한 ADR이 Accepted 상태임
- Human이 구현 시작을 승인함

READY 이후 요구사항이나 설계가 바뀌면 Codex가 임의 반영하지 않고 Issue를 다시 Refinement한다.

### 6단계: Codex 구현

- LOW: 기존 패턴을 따르는 명확한 작업이므로 구현부터 PR 생성까지 진행한다.
- MEDIUM: 짧은 구현 계획과 영향 범위를 Human에게 보고하고 승인 후 진행한다.
- HIGH: 필요한 ADR, 명시적 Human 설계 승인, 구현 계획 승인 후 진행한다.

Codex는 코드, 테스트, 로컬 검증에 집중한다. 설계 문서나 요구사항을 임의로 바꾸지 않는다.

### 7단계: 테스트·검증·트러블슈팅

Codex는 가장 작은 관련 테스트부터 실행하고, 전체 테스트와 빌드, diff 검증을 수행한다.

Issue와 PR에는 테스트 개수보다 다음을 기록한다.

- 테스트가 보장하는 것
- 테스트가 보장하지 않는 것
- 직접 검증 방법
- DB·로그·상태 확인 결과
- 미검증 항목

```text
테스트 = 코드가 예상대로 동작하는지 확인
검증 = 해결하려던 문제가 실제로 해결됐는지 확인
```

일반적인 컴파일 오류, import 누락, 단순 테스트 데이터나 Mock 설정 오류는 Codex가 자체 해결한다.

다음과 같은 중요한 문제는 Troubleshooting Gate에서 멈춘다.

- 요구사항 변경 없이는 테스트를 통과할 수 없음
- API 계약, DB 스키마, 트랜잭션 경계, 권한 정책 변경 필요
- 동시성 정합성, 데이터 손실, 중복 처리 가능성 발견
- 테스트 삭제·비활성화·약화가 필요함
- 성능 목표를 달성하지 못함
- ADR과 구현이 충돌함
- 임시 우회와 근본 해결 중 선택 필요
- Issue 범위 밖 수정 필요

중요 문제가 발생하면 다음 흐름으로 처리한다.

```text
Codex 중단
→ Human + ChatGPT 문제 검토
→ 필요하면 Issue 또는 ADR 수정
→ Human 재승인
→ Codex 작업 재개
```

예상 트러블슈팅과 실제 트러블슈팅은 구분한다. 실제 발생하지 않은 문제를 해결 사례로 기록하지 않는다.

### 8단계: Human 코드 확인

구현이 끝나면 Human은 최소한 다음을 직접 설명할 수 있어야 한다.

- 요청과 호출 흐름
- 핵심 규칙이 적용되는 위치
- 트랜잭션 시작과 종료
- 실패 시 데이터 상태
- 테스트가 증명하는 것
- 직접 검증한 내용
- 선택한 기술과 트레이드오프
- 아직 검증하지 못한 부분

설명할 수 없는 코드는 그대로 Merge하지 않는다.

### 9단계: Commit·Push·PR

검증이 끝나면 Codex가 별도 단계별 승인 없이 다음을 수행한다.

1. Issue 범위의 코드와 테스트를 Commit한다.
2. 작업 브랜치를 Push한다.
3. PR을 생성한다.
4. PR 번호를 Human에게 전달한다.

PR에는 구현 내용, 테스트 결과, 보장 범위, 검증 결과, 트러블슈팅, 미검증 항목, ChatGPT 인계 요청을 기록한다.

### 10단계: PR 기반 인계와 리뷰

Human은 ChatGPT에 PR 번호만 전달한다.

```text
PR #<번호> 검토하자.
```

ChatGPT는 GitHub에서 다음을 직접 확인한다.

- 관련 Issue와 Human 사전 검토
- 관련 ADR
- 변경 파일과 실제 diff
- 커밋 범위
- GitHub Actions 상태와 실패 로그
- 테스트가 완료 조건을 검증하는지
- 범위 이탈과 문서 불일치
- 트랜잭션, 정합성, 동시성, 보안, 성능 위험
- 구현자가 이해해야 할 핵심 흐름

리뷰는 자동 승인 절차가 아니라 Human이 코드를 이해하고 반영 여부를 판단하는 시간이다.

중요 PR은 필요하면 다른 팀원 또는 독립 Codex Reviewer의 검토를 추가한다. 모든 PR에 동일한 다중 리뷰를 강제하지 않는다.

### 11단계: 수정과 재검증

수정이 필요하면 ChatGPT와 Human이 다음을 명확히 정리한다.

- 반영할 리뷰와 이유
- 반영하지 않을 리뷰와 이유
- 수정 허용·금지 범위
- 추가 테스트와 검증
- 완료 조건

Codex는 같은 브랜치와 같은 PR에 승인된 수정만 반영하고 다시 검증한다.

### 12단계: Human Merge

Merge 조건:

- Issue 완료 조건 충족
- Actions 성공
- 테스트와 검증 결과 확인
- 중요한 트러블슈팅 해결
- ChatGPT 리뷰 완료
- Human이 코드를 이해함
- Human 최종 승인

절대 규칙:

```text
ChatGPT Merge 금지
Codex Merge 금지
자동 Merge 금지
Human만 Merge 가능
```

AI가 Merge 승인 문구를 남기더라도 실제 Merge 권한과 실행은 Human에게만 있다.

### 13단계: 구현 후 문서 보완

모든 PR마다 모든 문서를 갱신하지 않는다.

- API 변경 → API 명세
- DB 구조 변경 → ERD
- 중요한 선택 → ADR
- 실행 방법 변경 → README
- 실제로 발생한 의미 있는 문제 해결 → 트러블슈팅 문서
- 중요한 AI 리뷰 판단 → AI Review Log

구현 결과에 종속되는 문서는 ChatGPT가 PR 검토 후 별도 문서 PR로 보완할 수 있다.

## 4. 리뷰 강도

- 일반 PR: ChatGPT 리뷰 + Human 직접 검토
- 중요 PR: ChatGPT 리뷰 + 필요 시 팀원 또는 독립 Codex Review + Human 직접 검토
- 문서 PR: 내용 정합성 확인 + Actions

모든 PR에 동일한 수준의 다중 리뷰와 로그를 강제하지 않는다. 실제 위험과 학습 가치에 따라 리뷰 강도를 조정한다.

## 5. 성공 기준

이 워크플로우는 자동화 단계 수가 아니라 다음 결과로 평가한다.

- 구현 시간이 줄어드는가
- 토큰과 반복 전달이 줄어드는가
- Human이 문제와 설계를 직접 설명할 수 있는가
- Human이 필요한 판단에 집중하는가
- 테스트와 검증 품질이 유지되거나 좋아지는가
- 중요한 상황에서만 작업이 중단되는가
- 개발자가 결과와 트레이드오프를 설명할 수 있는가
- 실패와 트러블슈팅이 사실대로 남는가
