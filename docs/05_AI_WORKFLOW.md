# Team OASIS AI 활용 워크플로우

## 1. 목적

최종 프로젝트에서 AI를 많이 사용하는 것 자체는 성과가 아니다.

이 워크플로우의 목적은 다음과 같다.

- 팀원 모두가 자신이 구현한 코드와 테스트를 설명할 수 있게 한다.
- AI가 잘못된 코드를 만들더라도 사람과 테스트가 걸러낼 수 있게 한다.
- 도메인 설계, 트랜잭션, 데이터 정합성, 테스트, 성능 개선을 프로젝트의 중심에 둔다.
- 반복 지시와 전달 비용은 줄이되 중요한 Human 판단은 유지한다.
- AI 자동화 구축 때문에 실제 백엔드 개발 시간이 줄어들지 않게 한다.

성과 비중은 다음을 지향한다.

```text
백엔드 문제 해결 70%
테스트·협업·문서화 20%
AI 활용과 통제 방식 10%
```

AI 하네스는 프로젝트의 주인공이 아니라 백엔드 결과물을 더 안전하고 일관되게 만들기 위한 협업 방식이다.

## 2. 기본 역할

### Human

- 서비스 목적과 우선순위 결정
- 요구사항과 도메인 규칙 해석
- API, ERD, 트랜잭션, 권한, 상태 전이 설계 승인
- Issue 착수 전 문제·범위·흐름·실패 시나리오·검증 계획 작성
- 위험도와 리뷰 수준 최종 결정
- 중요한 기술 선택과 ADR 승인
- AI가 작성한 코드와 테스트 이해
- Troubleshooting Gate 선택지 결정
- 최종 리뷰와 Merge 직접 수행
- 결과 책임

### ChatGPT

- 프로젝트 문서와 API·ERD를 기반으로 전체 Draft Issue 초안 작성
- 담당자의 착수 검토를 문서·현재 코드와 비교해 누락·충돌·위험·대안 제시
- Human 결정을 반영한 Issue, ADR, Markdown 정리
- PR의 실제 diff, 커밋, Actions, 테스트 증거 검토
- 구현 결과에 종속되는 문서 보완 지원
- 중요한 충돌을 발견하면 임의로 맞추지 않고 Human에게 보고

### Codex

- 승인된 `READY` Issue 구현
- 코드와 테스트 작성
- 로컬 실행, 디버깅, 테스트, 빌드, diff 검증
- Issue 범위 안의 일반 오류 자체 해결
- 실제 diff 자체 리뷰와 재검증
- Commit, 작업 브랜치 Push, Draft PR 생성
- 승인된 리뷰 수정 사항을 같은 PR에 반영
- 설계·정합성·보안·성능 문제가 발견되면 중단하고 보고

### GitHub

- Issue로 요구사항, 완료 조건, 상태 관리
- API·ERD·ADR·문서로 설계 근거 보존
- Branch와 Commit으로 구현 이력 관리
- Draft PR로 구현 결과 인계
- Actions로 테스트와 빌드 결과 제공
- 리뷰와 Human Merge 이력 보존

## 3. 프로젝트 시작 기준선

프로젝트 시작 시 다음 기준선을 준비한다.

- `AGENTS.md`
- 프로젝트 목표와 요구사항 문서
- API 명세
- ERD와 Migration 기준
- `EVIDENCE_GUIDE`
- ADR 기준과 템플릿
- Issue / PR / Code Review 템플릿
- 공통 테스트·빌드 검증 명령
- GitHub Actions
- 보호 브랜치 직접 Push를 막는 최소 Hook
- Human-only Merge 정책

이 문서들은 AI에게 모든 판단을 맡기기 위한 것이 아니라 팀원과 AI가 같은 기준으로 작업하기 위한 기준선이다.

## 4. 전체 Workflow

```text
프로젝트 기준선 준비
→ 전체 Draft Issue 초안 생성
→ 팀 백로그 검토
→ 담당자 착수 검토
→ AI 재검증
→ Human 승인과 READY 전환
→ Codex Read / Plan / Implement / Verify / Review Diff / Fix / Re-verify / Report
→ Commit / Push / Draft PR
→ 위험도별 리뷰와 CI
→ Human Merge
→ 배포·통합 확인과 필요한 문서 갱신
```

### 4.1 전체 Draft Issue 초안

AI가 프로젝트 문서, API, ERD를 근거로 전체 개발 Issue 초안을 만든다.

목적은 AI에게 프로젝트를 맡기는 것이 아니라 전체 개발 지도를 먼저 확인하는 것이다.

초기에 확인할 내용:

- 필수 기능 누락 여부
- Issue 선후 관계와 의존성
- API·ERD·Migration 선행 작업
- 팀원별 작업량과 병렬 가능 범위
- 5주 안에 가능한 범위
- 트랜잭션·동시성·성능 검증이 필요한 기능
- 불필요한 기술과 도전 과제의 우선순위

초기 Issue는 확정 설계가 아니라 `DRAFT`다.

```text
DRAFT
→ READY
→ IN_PROGRESS
→ REVIEW
→ DONE
```

### 4.2 팀 백로그 검토

팀은 전체 Draft Issue를 검토한다.

- 기능 누락
- Issue 의존관계
- 작업 크기와 우선순위
- 팀원별 분배
- 잠정 위험도
- 필수 기능보다 도전 기술이 앞서는지
- 일정 안에 완성 가능한지

### 4.3 담당자 착수 검토

실제 구현 담당자는 AI Draft를 그대로 승인하지 않고 관련 문서와 현재 코드를 읽은 뒤 자신의 언어로 작성한다.

- 내가 이해한 문제
- 왜 지금 필요한지
- 포함 범위와 제외 범위
- 적용되는 도메인 규칙
- 예상 코드·데이터 흐름
- 주요 정상·실패·경계 시나리오
- 고려한 대안과 트레이드오프
- 테스트·직접 검증 계획
- 예상 트러블슈팅
- 확신하지 못해 검토가 필요한 부분

LOW 작업은 간단히 작성할 수 있다. 금액, 권한, 상태 전이, 트랜잭션, 동시성, 캐시, 이벤트처럼 중요한 작업은 실패 시 데이터 상태까지 구체적으로 작성한다.

### 4.4 AI 재검증

ChatGPT는 담당자 착수 검토를 프로젝트 docs, API, ERD, ADR, 기존 코드와 비교한다.

- 범위가 과하거나 모호하지 않은가
- 도메인 규칙과 실패 시나리오가 누락되지 않았는가
- API·ERD·Migration과 충돌하지 않는가
- 위험도가 적절한가
- 테스트가 실제 규칙을 증명하는가
- 테스트 외 직접 검증이 필요한가
- 불필요한 기술이나 구조가 들어가지 않았는가
- 중요한 설계 선택에 ADR이 필요한가

AI의 제안은 자동 확정안이 아니다. Human이 요구사항, 범위, 설계, 위험도를 최종 결정한다.

### 4.5 READY 승인

다음 조건을 통과해야 Issue가 `READY`가 된다.

- 담당자가 Issue를 자신의 언어로 해석함
- 문제, 규칙, 흐름, 실패 시나리오, 테스트·검증을 검토함
- AI 재검증 후 Human 최종 결정이 기록됨
- 완료 조건이 테스트하거나 검증 가능함
- 제외 범위와 수정 경계가 명확함
- 선행 Issue가 완료됨
- API·ERD·설정 영향이 확인됨
- 위험도와 리뷰 수준이 결정됨
- 필요한 ADR이 Accepted 상태임
- Human이 구현 시작을 승인함

READY 이후 설계가 바뀌면 Codex가 임의 반영하지 않고 Issue를 다시 Refinement한다.

### 4.6 Codex 구현

주 Codex가 다음 순서로 수행한다.

```text
Read
→ Plan
→ Implement
→ Verify
→ Review Diff
→ Fix
→ Re-verify
→ Report
```

- `Read`: Issue, Human 결정, 관련 문서와 코드만 읽는다.
- `Plan`: 구현 범위, 예상 파일, 순서, 테스트, 제외 범위를 정리한다.
- `Implement`: Issue 범위 안에서 최소 변경한다.
- `Verify`: 관련 테스트, 필요한 통합·동시성·성능 테스트, 전체 테스트, 빌드, diff 검증을 실행한다.
- `Review Diff`: 실제 diff를 다시 읽어 범위 이탈, 핵심 버그, 테스트 누락을 확인한다.
- `Fix`: Issue 범위 안의 수정 가능한 문제만 해결한다.
- `Re-verify`: 수정 후 관련 검증을 다시 수행한다.
- `Report`: 실행한 명령과 결과, 테스트 보장·비보장 범위, 직접 검증, 미검증 항목을 기록한다.

복잡한 Agent Teams와 상시 Subagent 분리는 기본값으로 사용하지 않는다. 독립 리뷰는 위험도에 따라 별도 단계로 수행한다.

### 4.7 Troubleshooting Gate

다음 상황에서는 Codex가 임의 진행하지 않는다.

- 문서와 실제 코드가 충돌함
- Issue 범위를 확장해야 함
- API·ERD·DB 스키마 변경 필요
- 인증·권한·트랜잭션 설계 재결정 필요
- 동시성 정합성, 데이터 손실, 중복 처리 위험 발견
- 테스트를 삭제·비활성화·약화해야 함
- 성능 목표 미달 또는 운영 장애 가능성 발견
- ADR과 구현이 충돌함
- 임시 우회와 근본 해결 중 선택 필요

처리 흐름:

```text
Codex 중단
→ 확인된 사실과 추정 원인 분리 보고
→ Human + ChatGPT 검토
→ 필요하면 Issue·API·ERD·ADR 수정
→ Human 재승인
→ Codex 작업 재개
```

### 4.8 Commit·Push·Draft PR

검증이 끝난 Codex는 작업 브랜치에 다음을 수행한다.

1. Issue 범위의 변경만 Commit
2. 작업 브랜치 Push
3. Draft PR 생성
4. PR 번호를 Human에게 전달

Draft PR에는 다음을 기록한다.

- 관련 Issue와 위험도
- Human 최종 설계 결정 요약
- 구현 내용과 주요 변경 파일
- 실제 실행한 테스트·빌드 명령과 결과
- 테스트가 보장하는 것과 보장하지 않는 것
- 직접 검증 결과
- 실제 발생한 의미 있는 트러블슈팅
- 미검증 항목과 알려진 제한
- 독립 리뷰 필요 여부
- 작성자가 설명해야 할 핵심 흐름
- 필요한 문서 보완

### 4.9 위험도별 리뷰

#### LOW

예: 문서, 오타, 단순 CRUD, 국소 DTO 변경.

- Codex 자체 diff 리뷰
- 작성자 Human 리뷰
- 관련 테스트와 CI

#### MEDIUM

예: 서비스 비즈니스 로직, 새로운 API, 조회 쿼리, 인덱스, 캐시, 이벤트, 여러 계층 영향.

- 정상·실패·경계 테스트
- Codex 자체 diff 리뷰
- 필요 시 독립 AI 또는 다른 팀원 리뷰
- 작성자 Human 리뷰
- CI

#### HIGH

예: 인증·인가, 금액·포인트·재고, 상태 전이, 트랜잭션, 동시성, DB 구조 변경.

- 구현 전 설계 확인과 필요한 ADR
- 권한·Rollback·동시성·정합성 테스트
- 독립 AI 리뷰
- 강화된 Human 리뷰
- CI

모든 PR에 ChatGPT·Codex·독립 Codex 3중 리뷰를 강제하지 않는다. 위험도와 학습 가치에 따라 검증 비용을 조정한다.

### 4.10 Merge

Merge 전 Human은 다음을 확인한다.

- Issue 완료 조건 충족
- 최신 Commit 기준 GitHub Actions 성공
- 위험도에 맞는 리뷰 완료
- 문서와 코드 충돌 없음
- 범위 밖 변경 없음
- 테스트 삭제·약화 없음
- 미검증 항목과 알려진 제한 확인
- 작성자가 코드, 테스트, 실패·Rollback 흐름을 설명 가능

절대 규칙:

```text
ChatGPT Merge 금지
Codex Merge 금지
자동 Merge 금지
Human만 Merge 가능
```

### 4.11 Merge 이후

필요한 항목만 수행한다.

- 배포·통합 환경 확인
- API 변경 시 API 명세 갱신
- DB 구조 변경 시 ERD·Migration 문서 갱신
- 실제로 발생한 의미 있는 문제는 트러블슈팅 기록
- 성능 개선은 적용 전후 결과 기록
- 중요한 설계 결정은 ADR 기록
- 오래되거나 충돌하는 AI 지침 정리

모든 PR마다 모든 문서와 AI 로그를 갱신하지 않는다.

## 5. 적용하지 않을 자동화

이번 최종 프로젝트에서는 다음을 기본으로 사용하지 않는다.

- Issue부터 Merge까지 완전 자동화
- 모든 Issue의 다중 Subagent 분리
- 모든 PR의 3중 AI 리뷰
- AI 자동 Merge
- 무제한 자동 수정 반복
- 모든 AI 실행 로그 저장
- 복잡한 Agent Teams와 Worktree 병렬 구현
- 하네스 자체를 프로젝트의 핵심 기술로 발표

제외 이유는 가치가 없어서가 아니라 4명의 신입 개발자가 5주 동안 유지하기에는 설정·운영 비용이 백엔드 개발 효과보다 커질 수 있기 때문이다.

## 6. Team OASIS AI 레인저스 역할

모든 팀원은 백엔드 기능을 개발하며 다음 역할은 프로젝트 공통 품질을 관리하는 추가 책임이다.

### AI 작전참모

- `AGENTS.md`와 전체 AI 작업 흐름 관리
- 문서 라우팅과 Issue→구현→검증→PR 흐름 점검
- 주요 정책과 Skill 도입 여부 판단
- AI 활용이 프로젝트 목적보다 커지지 않도록 조정

> 작전은 AI가 제안하지만 최종 명령은 인간이 내린다.

### AI 환각 검수원

- 존재하지 않는 코드·설정·테스트 결과 주장 확인
- 실행하지 않은 테스트를 통과했다고 보고하는지 검수
- Issue, API, ERD, 코드, 증거 일치 확인
- HIGH PR 독립 AI 리뷰 운영

> 그 테스트 진짜 돌린 거 맞습니까?

### AI 회유 담당

- Issue 범위 밖 리팩터링과 과잉 구현 차단
- 불필요한 기술과 갑작스러운 구조 변경 통제
- 문서 충돌 시 중단·보고하도록 확인
- 보호 브랜치 직접 작업, 자동 Merge, 위험 Git 명령 방지

> 좋아, 진정하고 Issue 범위 안으로 돌아가자.

### AI 재교육 담당

- 반복 실패 원인 분석
- 반복 문제가 확인될 때만 AGENTS, 템플릿, Hook, Skill 개선
- 하네스 드라이런과 오래된 지침 정리
- 프롬프트 추가보다 기준선과 검증 구조 개선 우선

> 또 틀렸으면 프롬프트가 아니라 교육과정을 바꾼다.

담당자가 관련 작업을 대신하는 것은 아니다. 각 기능 담당자는 자신의 코드, API, 문서, 테스트를 직접 책임진다.

## 7. 성공 기준

이 워크플로우는 자동화 단계 수가 아니라 다음 결과로 평가한다.

- 반복 지시와 AI 사이의 전달 비용이 줄었는가
- 구현 시간이 하네스 운영 때문에 줄지 않았는가
- 담당자가 문제와 설계를 직접 설명할 수 있는가
- 테스트와 직접 검증의 품질이 좋아졌는가
- 위험한 작업에만 강화 리뷰가 적용되는가
- AI의 범위 이탈과 허위 완료 보고를 발견할 수 있는가
- 실패와 미검증 항목이 사실대로 남는가
- 최종 백엔드 성과가 AI 활용 방식보다 앞에 있는가
