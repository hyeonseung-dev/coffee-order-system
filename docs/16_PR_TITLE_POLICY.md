# PR 제목 작성 정책

## 1. 기본 원칙

Pull Request 제목은 작업 내용을 한국어로 바로 이해할 수 있도록 작성한다.

- PR 제목의 핵심 작업명은 반드시 한국어로 작성한다.
- 제목 전체를 영어로만 작성하지 않는다.
- 클래스명, 메서드명, API, Redis, MySQL, Kafka처럼 번역하면 오히려 의미가 흐려지는 기술 식별자는 그대로 사용할 수 있다.
- 제목만 읽어도 무엇을 추가·수정·검증했는지 알 수 있어야 한다.
- Issue 제목과 실제 변경 범위가 다르면 실제 변경 범위를 기준으로 작성한다.

## 2. 권장 형식

```text
분류: 한국어 작업 요약
```

권장 분류:

- `기능`: 새로운 기능 추가
- `수정`: 버그 또는 잘못된 동작 수정
- `테스트`: 테스트 추가·보완
- `문서`: 문서와 작업 규칙 변경
- `설정`: 빌드·환경·운영 설정 변경
- `리팩터링`: 동작 변경 없는 구조 개선

## 3. 예시

좋은 예:

```text
테스트: 동일 사용자 동시 주문 정합성 재현
문서: 실제 변경 기반 질문 난이도 정책 추가
수정: 포인트 지갑 조회에 비관적 락 적용
설정: MySQL 동시성 테스트 실행 환경 분리
```

피해야 할 예:

```text
test: observe concurrent order consistency
docs: enforce junior-level Diff questions
fix wallet concurrency issue
```

## 4. Draft PR 생성 전 확인

Codex는 Draft PR을 만들기 전에 다음을 확인한다.

- [ ] 제목의 핵심 작업명이 한국어다.
- [ ] 영어 전용 제목이 아니다.
- [ ] 제목이 실제 Diff와 일치한다.
- [ ] 제목만으로 변경 목적을 파악할 수 있다.

조건을 충족하지 못하면 PR을 생성하거나 보고하기 전에 제목을 수정한다.
