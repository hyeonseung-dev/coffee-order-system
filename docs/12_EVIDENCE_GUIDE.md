# Evidence Guide

이 문서는 테스트·검증·리뷰 결과를 사실대로 남기기 위한 최소 기준이다. 목적은 형식적인 로그를 늘리는 것이 아니라, PR의 주장과 실제 근거를 연결하는 것이다.

## 1. 기본 원칙

- 실행하지 않은 명령을 실행했다고 기록하지 않는다.
- 테스트 통과만으로 요구사항이 해결됐다고 단정하지 않는다.
- 확인된 사실, 추정, 가정, 미검증 항목을 구분한다.
- 로컬 결과와 GitHub Actions 결과를 혼동하지 않는다.
- 테스트가 보장하는 것과 보장하지 않는 것을 함께 기록한다.
- 실제로 발생하지 않은 문제를 트러블슈팅 사례로 만들지 않는다.
- 성능·안정성·보안 개선은 측정값이나 재현 가능한 근거 없이 주장하지 않는다.

## 2. Issue에서 정의할 증거

READY 전 Issue에는 최소한 다음이 있어야 한다.

- 핵심 불변식 또는 도메인 규칙
- 정상·실패·경계 시나리오
- 필요한 테스트 종류
- 직접 확인해야 하는 API·DB·로그·상태
- 성공 기준
- 허용 가능한 미검증 범위
- 문제가 발생하면 Human에게 반환할 조건

예시:

```text
규칙: 잔액 부족 주문은 실패하며 잔액, 주문, 포인트 이력이 모두 변경되지 않는다.
테스트: Rollback 통합 테스트
직접 검증: 실패 요청 전후 point_wallet, orders, point_history 비교
성공 기준: 세 저장 상태 모두 변경 없음
```

## 3. PR에 반드시 남길 증거

### 실행한 명령과 결과

```text
./gradlew test        : 성공 / 실패 / 미실행
./gradlew build       : 성공 / 실패 / 미실행
git diff --check     : 성공 / 실패 / 미실행
직접 API 검증         : 내용 / 미실행 사유
DB·로그 확인          : 내용 / 미실행 사유
```

명령의 전체 출력 복사보다 실행 명령, 결과, 실패 시 핵심 원인을 기록한다.

### 테스트가 보장하는 것

- 어떤 규칙을 어떤 테스트가 검증하는지 연결한다.
- 성공 경로뿐 아니라 실패·Rollback·권한·경계 조건을 포함한다.

### 테스트가 보장하지 않는 것

예:

- 실제 다중 서버 환경
- 운영 트래픽 수준
- 네트워크 단절
- 외부 시스템 장애
- Replica 지연
- Redis failover

검증하지 않은 범위를 성공한 것처럼 표현하지 않는다.

### 직접 검증

테스트 외에 필요한 경우 다음을 확인한다.

- HTTP 요청과 응답
- DB 최종 상태
- 트랜잭션 실패 후 데이터 상태
- 로그와 이벤트 발생 여부
- 캐시 Hit/Miss와 TTL
- 실행계획과 측정값

### 미검증 항목과 제한

미검증은 실패가 아니다. 다만 숨기지 않고 후속 작업 또는 한계로 명시한다.

## 4. 위험도별 최소 증거

### LOW

- 관련 테스트
- 전체 CI
- diff 확인
- 범위 이탈 없음

### MEDIUM

LOW 기준에 추가:

- 정상·실패·경계 테스트
- API·쿼리·캐시·이벤트 등 변경 영향 확인
- 직접 검증 결과
- 필요 시 독립 AI 또는 다른 팀원 리뷰 근거

### HIGH

MEDIUM 기준에 추가:

- 구현 전 설계·ADR·Human 승인
- 권한·Rollback·동시성·정합성 등 해당 위험 테스트
- 독립 AI 리뷰
- 강화된 Human 리뷰
- 실패 시 데이터 상태와 복구·재시도 한계

## 5. 성능 개선 증거

성능 개선 PR에는 최소한 다음을 남긴다.

- 동일한 데이터와 조건
- 적용 전 실행계획 또는 측정값
- 적용 후 실행계획 또는 측정값
- 측정 명령과 횟수
- 평균·대표값과 편차
- 개선된 지표
- 악화된 비용: 쓰기, 저장 공간, 복잡도 등
- 측정 환경의 한계

단일 실행 결과만으로 일반적인 성능 개선을 단정하지 않는다.

## 6. 트러블슈팅 증거

의미 있는 문제가 실제로 발생했을 때만 기록한다.

```markdown
## 발생한 문제

## 재현 조건

## 확인된 사실

## 추정 원인

## 시도한 방법과 결과

## 실제 원인

## 최종 해결

## 재발 방지 테스트

## 남은 한계
```

확인된 사실과 추정 원인을 섞지 않는다. 해결하지 못한 경우 해결한 것처럼 쓰지 않고 `BLOCKED`와 필요한 Human 결정을 남긴다.

## 7. 리뷰 증거

리뷰 의견은 다음 구조를 우선한다.

- 위치
- 문제
- 위험
- 근거
- 수정 방향
- 추가 테스트 또는 재검증 방법

단순 취향이나 스타일보다 요구사항, 데이터 정합성, 트랜잭션, 권한, 동시성, 성능, 운영 장애 가능성을 우선한다.

AI 리뷰는 자동 승인 근거가 아니다. Human이 실제 diff와 증거를 확인하고 반영 여부를 결정한다.

## 8. Merge 증거

Merge 전 Human은 다음을 확인한다.

- Issue 완료 조건 충족
- 최신 Commit 기준 GitHub Actions 성공
- 요구된 리뷰 완료
- 문서와 코드 충돌 없음
- 범위 밖 변경 없음
- 작성자가 코드와 테스트를 설명 가능
- 미검증 항목과 알려진 제한 확인

ChatGPT, Codex, 자동화는 Merge하지 않는다.
