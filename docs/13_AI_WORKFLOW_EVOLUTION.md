# AI Issue Refinement Workflow Evolution

## 1. 문서 목적

이 문서는 개인 프로젝트에서 AI-assisted 개발 워크플로우를 실제로 적용하면서 발견한 문제와 개선 과정을 기록한다.

단순히 최종 규칙만 남기는 것이 아니라 다음을 보존한다.

- 어떤 방식으로 시작했는가
- 실제 적용 중 무엇이 비효율적이거나 위험했는가
- Human과 AI의 역할을 어떻게 다시 나눴는가
- 최종 프로젝트에서 재사용할 운영 원칙은 무엇인가
- 포트폴리오에서 어떤 문제 해결 경험으로 설명할 수 있는가

구체적인 PR·Issue별 사건 기록은 `docs/10_AI_REVIEW_LOG.md`에 남기고, 이 문서는 반복 적용할 수 있는 일반화된 프로세스를 설명한다.

---

## 2. 기존 방식과 문제

초기 방식은 AI가 Issue 제목과 기본 항목을 만들고, 담당 Human이 다음 내용을 대부분 직접 채우는 구조였다.

```text
AI가 빈 양식 중심의 Draft Issue 작성
→ Human이 문제·흐름·대안·테스트를 거의 처음부터 작성
→ AI 재검증
→ Human 승인
→ Codex 구현
```

이 방식은 Human이 생각하도록 만든다는 장점이 있었지만 실제 적용에서는 다음 문제가 발생했다.

### 2.1 AI 초안이 지나치게 빈약함

AI가 제목과 `작성 필요` 항목만 제공하면 Human은 AI의 설계안을 검토하는 것이 아니라 설계를 처음부터 작성해야 한다.

이는 다음 문제로 이어졌다.

- Issue 작성 시간이 구현 시간보다 길어질 수 있음
- 학습 검증이 아니라 빈 양식 채우기가 됨
- 아직 배우지 않은 구현 세부사항까지 Human이 추측해야 함
- AI가 제공할 수 있는 문서·코드 분석과 선택지 발굴 기능을 활용하지 못함

### 2.2 반대로 AI가 Human 판단까지 대신할 위험

Issue #9 착수 과정에서는 AI가 Human 작성이 끝나지 않았는데도 정책을 확정하고 `READY`로 전환하는 오류가 발생했다.

이 경우 Human-in-the-loop가 형식만 남고, 실제로는 AI가 다음을 대신하게 된다.

- 도메인 정책 결정
- 구현 방식 확정
- 테스트 기준 확정
- 구현 시작 승인

따라서 AI 초안을 상세하게 만드는 것과 Human 결정을 대체하는 것은 구분해야 한다.

### 2.3 재검증 단계가 계속 늘어날 위험

`AI 재검증 1차 → 2차 → 3차`를 반복하면 정확도는 높아질 수 있지만 프로젝트의 중심이 구현과 학습이 아니라 절차 관리로 이동할 수 있다.

이 프로젝트에서는 공식 AI 재검증을 한 번만 기록한다. 이후에는 Human과 AI가 대화로 협의하고, 최종 합의된 Issue를 한 번 정리한다.

## 3. 개선된 원칙

### 핵심 원칙

```text
AI는 판단 가능한 상세 설계안을 제공한다.
Human은 자신의 언어로 흐름·대안·트레이드오프를 설명한다.
AI 재검증은 공식적으로 한 번만 수행한다.
이후 쟁점은 대화로 협의한다.
Codex는 최종 합의된 READY Issue만 구현한다.
```

역할 비중은 다음을 지향한다.

```text
AI Draft: 요구사항·현재 코드·권장안·대안·경계 시나리오의 70~80%
Human Refinement: 이해 검증·누락 발견·도메인 정책 결정의 20~30%
```

이 비율은 Human의 판단을 줄이기 위한 것이 아니다. Human이 빈 문서 작성에 시간을 쓰지 않고, 실제로 중요한 선택과 검증에 집중하기 위한 것이다.

## 4. 최종 Issue Refinement 흐름

```text
1. AI 상세 Draft Issue 작성
2. Human 담당자 착수 검토 작성
3. AI 재검증 1차
4. Human과 AI가 대화로 쟁점 협의
5. AI가 최종 합의 Issue로 정리
6. Human이 최종 확인 후 READY 승인
7. Codex가 구현·검증·Draft PR 생성
8. 위험도에 맞는 리뷰 후 Human Merge
```

### 4.1 AI 상세 Draft Issue

AI는 프로젝트 문서, 현재 코드, 선행 Issue를 읽고 다음을 먼저 작성한다.

- 작업 목적과 현재 필요한 이유
- 현재 코드와 문서에서 확인한 사실
- AI 권장 API와 도메인 정책
- 예상 코드·데이터 흐름
- 고려한 대안과 트레이드오프
- 정상·실패·경계 시나리오
- 테스트 및 직접 검증 초안
- 포함·제외 범위
- Human이 결정해야 하는 항목

AI는 선택지를 단순 나열하지 않고 권장안과 근거를 제시한다. 다만 요구사항으로 확정되지 않은 내용은 `AI 권장안` 또는 `Human 결정 필요`로 표시한다.

### 4.2 Human 담당자 착수 검토

Human은 AI 초안을 복사하지 않고 자신의 언어로 다음을 작성한다.

- 내가 이해한 문제
- 내가 생각한 코드·데이터 흐름
- 내가 고려한 대안과 트레이드오프
- AI 권장안에 동의하는 부분
- 수정하거나 추가할 부분
- 정상·실패·경계 시나리오
- 테스트·직접 검증 계획
- 아직 이해되지 않는 부분

Human의 핵심 역할은 구현 의사코드를 처음부터 작성하는 것이 아니라 다음을 증명하는 것이다.

- 기능의 목적과 데이터 원본을 이해했는가
- 책임 분리와 트랜잭션 흐름을 설명할 수 있는가
- AI가 누락한 정책을 발견할 수 있는가
- 대안의 장단점을 비교하고 최종 결정을 내릴 수 있는가

### 4.3 AI 재검증 1차

AI는 Human 작성 내용을 현재 코드와 문서에 대조해 한 번만 공식 재검증한다.

- 잘 이해한 부분
- 수정이 필요한 개념
- 누락된 도메인 정책
- 현재 코드와 충돌하는 내용
- 불필요한 범위 확장
- 권장 수정안과 근거

AI는 Human 작성 영역을 덮어쓰지 않는다. Human의 최초 판단은 학습 증거로 보존하고, 별도의 재검증 영역에 의견을 기록한다.

### 4.4 대화 협의와 최종 합의

1차 재검증 이후에는 새로운 검증 섹션을 계속 추가하지 않는다.

Human과 AI가 채팅에서 다음을 협의한다.

- 이해되지 않는 기술
- 도메인 경계와 예외 정책
- API 계약
- 책임 분리
- 트랜잭션과 정합성
- 테스트 성공 기준
- 이번 Issue와 후속 Issue의 범위

협의가 끝나면 AI가 Issue를 최종 계약 형태로 정리한다.

### 4.5 Human READY 승인

최종 Issue에 다음이 확정되어야 한다.

- API와 요청·응답
- 도메인 규칙
- 정상·실패·경계 정책
- 코드·데이터 흐름
- 고려한 대안과 최종 선택 이유
- 구현 범위와 수정 금지 범위
- 테스트와 직접 검증 성공 기준
- 위험도와 리뷰 수준

마지막 `DRAFT → READY` 전환은 Human만 수행하거나 Human의 명시적 승인 후 반영한다.

## 5. Issue에서 미리 결정해야 하는 도메인 정책

모든 정책을 프로젝트 시작 전에 완벽하게 결정할 수는 없다. 따라서 각 Issue 착수 시 해당 기능의 결과를 바꾸는 정책을 구체화한다.

예시:

- 기간의 시작·종료 경계
- 정확히 경계값인 데이터의 포함 여부
- 1차·2차 정렬 기준
- 동률 처리
- 데이터가 없을 때 빈 목록·기본값·예외 중 무엇을 반환할지
- 주문 수가 0인 대상도 결과에 포함할지
- 성공·실패·취소 상태 중 집계 대상
- 중복 요청과 멱등성
- 최대 개수보다 데이터가 적을 때 처리

이런 선택은 다른 도메인까지 과도하게 확장하지 않으면서도 현재 기능의 결과를 결정하므로 Issue 단계에서 Human이 확정한다.

반면 다음은 현재 Issue의 정확성에 필요하지 않다면 후속 Issue로 분리한다.

- 인덱스
- Redis 캐시·분산락
- 메시지 브로커
- Replica
- Outbox
- 대규모 공통 리팩터링

## 6. Issue #9에서 확인한 실제 사례

최근 7일 인기 메뉴 TOP 3 Issue에서 다음 문제를 확인했다.

1. AI 초안이 빈 양식에 가까워 Human이 집계 위치, 정렬, 빈 결과, 테스트를 거의 처음부터 추측해야 했다.
2. Human은 MySQL 주문 데이터가 원본이라는 점과 Redis를 후속 단계로 분리해야 한다는 점을 이해했다.
3. Human은 동률, 주문 없음, Repository와 Service 책임 분리 같은 누락 정책을 발견했다.
4. AI는 Human 작성이 끝나지 않았는데 정책을 대신 확정하고 READY로 전환했다가 이를 되돌렸다.
5. 최종적으로 AI 상세 초안, Human 이해 검토, AI 재검증 1회, 대화 협의, 최종 합의 Issue 구조로 개선했다.

이 사례는 AI의 장점과 위험을 동시에 보여준다.

- 장점: 코드와 문서를 빠르게 분석하고 다양한 경계·대안을 제시함
- 위험: Human의 암묵적 의도와 승인 절차를 추론해 대신 결정할 수 있음
- 개선: AI가 초안을 충분히 작성하되 Human 영역과 최종 승인권을 명시적으로 분리함

## 7. 포트폴리오 활용 문장

> 초기에는 AI가 생성한 Issue가 빈 템플릿에 가까워 개발자가 설계를 처음부터 작성해야 했고, 반대로 AI가 Human 검토를 대신하면 이해하지 못한 설계가 바로 구현되는 문제가 있었습니다. 이를 개선해 AI가 현재 코드와 문서를 근거로 상세 설계안·대안·경계 시나리오를 제시하고, 개발자가 코드 흐름과 트레이드오프를 자신의 언어로 검증한 뒤 한 번의 AI 재검증과 협의를 거쳐 최종 Issue를 확정하는 프로세스를 설계했습니다. 이를 통해 AI 활용 속도와 Human의 이해·책임을 함께 유지했습니다.
