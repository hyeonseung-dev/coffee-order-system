# AI Issue Refinement Workflow Evolution

## 1. 문서 목적

이 문서는 개인 프로젝트에서 AI-assisted 개발 워크플로우를 실제로 적용하면서 발견한 문제와 개선 과정을 기록한다.

단순히 최종 규칙만 남기는 것이 아니라 다음을 보존한다.

- 어떤 방식으로 시작했는가
- 실제 적용 중 무엇이 비효율적이거나 위험했는가
- Human과 AI의 역할을 어떻게 다시 나눴는가
- 반복 실패를 프롬프트 추가가 아니라 실행 Gate로 어떻게 개선했는가
- 최종 프로젝트에서 재사용할 운영 원칙은 무엇인가
- 포트폴리오에서 어떤 문제 해결 경험으로 설명할 수 있는가

구체적인 PR·Issue 사건은 `docs/10_AI_REVIEW_LOG.md`와 `docs/reviews/`에 남기고, 이 문서는 일반화된 프로세스를 설명한다.

---

## 2. 1단계: 빈 Issue 템플릿 중심 방식

초기 방식은 AI가 Issue 제목과 기본 항목을 만들고, 담당 Human이 문제·흐름·대안·테스트를 대부분 직접 채우는 구조였다.

```text
AI가 빈 양식 중심 Draft Issue 작성
→ Human이 설계를 거의 처음부터 작성
→ AI 재검증
→ Human 승인
→ Codex 구현
```

### 문제

- Issue 작성 시간이 구현 시간보다 길어질 수 있음
- 학습 검증이 아니라 빈 양식 채우기가 됨
- 배우지 않은 구현 세부사항까지 Human이 추측해야 함
- AI의 문서·코드 분석과 대안 발굴 능력을 활용하지 못함

### 개선

AI가 현재 코드와 문서를 근거로 다음을 포함한 상세 Draft를 먼저 작성하도록 변경했다.

- 권장안과 근거
- 대안과 트레이드오프
- 정상·실패·경계 시나리오
- 테스트·직접 검증 초안
- 포함·제외 범위
- Human 결정 필요 항목

---

## 3. 2단계: AI가 Human 판단까지 대신한 문제

Issue #9 착수 과정에서는 반대 문제가 발생했다. AI가 Human 작성이 끝나지 않았는데도 정책을 확정하고 Issue를 `READY`로 전환했다.

이 경우 Human-in-the-loop가 형식만 남고 AI가 다음을 대신하게 된다.

- 도메인 정책 결정
- 구현 방식 확정
- 테스트 기준 확정
- 구현 시작 승인

### 개선

```text
AI 상세 Draft
→ Human 최초 이해 작성 영역 보존
→ AI 재검증은 별도 영역에 기록
→ Human·AI 쟁점 협의
→ 최종 합의 계약 정리
→ Human만 READY 승인
```

AI 초안을 상세하게 만드는 것과 Human 결정을 대체하는 것을 분리했다.

---

## 4. 3단계: 반복 재검증 과도화 방지

`AI 재검증 1차 → 2차 → 3차`를 반복하면 프로젝트의 중심이 구현과 학습이 아니라 절차 관리로 이동한다.

### 개선

- 공식 AI 재검증은 한 번만 기록
- 이후 쟁점은 대화로 협의
- 마지막에 최종 합의와 근거를 한 번 정리

```text
AI 최초 제안
→ Human 의견
→ 최종 합의
→ 선택 근거
```

---

## 5. 4단계: 구현 전 이해와 구현 후 코드 이해 분리

Issue Refinement를 강화해도 구현 전에는 예상 코드·데이터 흐름만 검토할 수 있다. Codex 구현 과정에서는 다음이 구체화되거나 달라질 수 있다.

- 실제 클래스와 메서드
- Repository Query
- 트랜잭션 경계
- 테스트 Fixture
- 예외 처리 방식
- 리뷰 반영 후 최종 Diff

테스트와 리뷰가 통과해도 담당자가 실제 Diff를 설명하지 못하면 인지 부채가 남는다.

### 최초 개선안

Merge 전에 다음 일반 관점을 확인하는 이해도 Gate를 추가했다.

- 해결한 문제
- 핵심 규칙의 구현 위치
- 정상·실패 시 데이터 상태
- 위험한 경계 조건
- 테스트 보장·비보장 범위

그러나 이 단계에서는 MEDIUM 작업이 고정된 일반 5문항을 사용하고, 실제 Diff 기반 질문은 HIGH에서만 추가하도록 설계했다.

---

## 6. 5단계: READY 이후 추가 승인을 다시 묻는 문제

Issue #9가 Human 검토와 READY 승인을 완료했는데도 Codex가 MEDIUM 작업이라는 이유로 구현 계획과 작업 브랜치 생성 승인을 다시 요청했다.

### 원인

- `READY = 구현 시작 승인`과 `MEDIUM = 계획 후 Human 확인`이 동시에 존재
- 보호 브랜치에서는 작업하지 말라는 규칙만 있고 깨끗한 보호 브랜치에서 작업 브랜치를 자동 생성하라는 규칙이 부족

### 개선

```text
READY = 구현 시작 승인 완료
```

- LOW·MEDIUM·HIGH 모두 READY 이후 추가 계획 승인 없음
- 깨끗한 `main`·`develop`이면 `codex/issue-{번호}-{slug}` 브랜치 자동 생성
- 미커밋 변경이나 계약 충돌이 있을 때만 중단

이 개선으로 실행 프롬프트를 다음 한 문장으로 유지할 수 있게 했다.

```text
GitHub Issue #9를 구현해주세요.
```

## 7. 6단계: 도구 부재를 전체 작업 중단으로 해석한 문제

Codex는 `gh` CLI가 없다는 이유로 Commit·Push·Draft PR 전부를 보류했다.

### 원인

Commit·Push와 PR 생성 수단을 구분하지 않았다.

### 개선

- Commit과 Push는 표준 `git` 명령으로 수행
- Draft PR은 사용 가능한 GitHub 연동, API 또는 `gh` 사용
- PR 생성 수단이 전혀 없을 때만 Push 후 PR 생성 단계만 BLOCKED

도구 하나의 부재를 수행 가능한 단계까지 중단하는 이유로 사용하지 않도록 했다.

## 8. 7단계: PR #32에서 확인한 실행 충실도 실패

Issue #9는 상세한 최종 계약을 가진 상태에서 Codex가 구현했다. 핵심 JPQL 구조는 올바르게 구현됐지만 다음 누락이 PR 리뷰에서 발견됐다.

- `Asia/Seoul` 고정 대신 `Clock.systemDefaultZone()` 사용
- SQL 또는 `JdbcTemplate` 대신 Reflection Fixture 사용
- TOP 3 초과, 3개 미만, ACTIVE 0개, Controller 빈 배열 테스트 일부 누락
- PR 템플릿 전체 대신 자체 요약 본문 작성
- 실제 수행 전 Human·ChatGPT 리뷰 완료 체크
- MEDIUM 작업인데 실제 Diff 기반 이해도 질문 없음

### 8.1 왜 상세 Issue가 있어도 계약을 놓쳤는가

문제는 Issue의 정보량 부족이 아니었다.

1. Issue에는 최초 제안, Human 원문, 재검증, 최종 합의가 함께 보존돼 있었다.
2. Codex가 권위 있는 최종 영역만 구조적으로 추출하지 않았다.
3. 자체 리뷰는 `Issue 완료 조건 충족`이라는 단일 체크였고 계약별 증거 연결이 없었다.
4. 테스트는 작성된 시나리오만 검증하므로 누락된 시나리오는 실패할 수 없었다.
5. Service Clock Mock은 KST였지만 실제 설정 Bean은 검증하지 않아 운영 경로 불일치를 놓쳤다.
6. Reflection Fixture도 결과는 맞게 만들 수 있어 구현 방식 계약 위반을 행동 테스트가 잡지 못했다.

### 8.2 왜 PR 템플릿을 준수하지 않았는가

기존 Skill은 PR에 위험도와 검증 증거를 기록하라고만 했다.

다음 하드 규칙이 없었다.

- 베이스 브랜치의 최신 템플릿을 PR 생성 직전에 다시 읽기
- 모든 `##`·`###` 제목 보존
- 빈 섹션을 자체 요약으로 대체하지 않기
- 외부 리뷰와 CI를 선행 체크하지 않기
- PR 생성 후 본문을 다시 검증하기

또한 MEDIUM은 일반 5문항, HIGH만 실제 Diff 기반 추가 질문이라는 설계가 질문 축약을 허용했다.

따라서 이번 문제는 Codex의 실행 오류이면서 동시에 **계약 추적과 템플릿 보존을 강제하지 않은 루프 설계 문제**였다.

구체 기록: `docs/reviews/PR_32_AI_REVIEW.md`

## 9. 8단계: Contract Traceability Gate 도입

Issue를 더 길게 만들지 않고 최종 계약을 원자 항목으로 분해한다.

### 구현 전

| ID | 최종 계약 | 예상 구현 위치 | 필요한 증거 |
|---|---|---|---|
| C-01 | 결과 또는 구현 제약 | 클래스·메서드·Query·설정 | 테스트·직접 검증·문서 |

### 구현 후

| ID | 최종 계약 | 실제 코드 증거 | 테스트·직접 검증 증거 | 상태 |
|---|---|---|---|---|
| C-01 | 계약 | `path#symbol` | `TestClass#method` 또는 사유 | `PASS | HOLD | N/A` |

### 규칙

- 권위 있는 영역은 최종 합의, 최종 구현 계약, 완료 조건, 검증 계획, 포함·제외 범위다.
- 결과 요구사항뿐 아니라 시간대, JOIN 위치, Fixture 방식, 도입 금지 기술 등 구현 제약도 추적한다.
- 모든 계약에 실제 코드와 검증 증거를 연결한다.
- 한 항목이라도 HOLD면 완료 보고, Commit, Draft PR을 진행하지 않는다.
- 테스트 통과는 계약 전체 PASS를 대신하지 않는다.

---

## 10. 9단계: PR Template Fidelity Gate 도입

Draft PR 생성 직전에 Codex가 베이스 브랜치의 최신 PR 템플릿을 다시 읽는다.

- 모든 제목 보존
- 빈 섹션 축약 금지
- Contract Traceability 표 작성
- Codex가 직접 완료한 항목만 체크
- Human·ChatGPT·독립 리뷰·CI·이해도 Gate 선행 체크 금지
- PR 생성 후 실제 본문 재검사

PR 템플릿은 참고 문서가 아니라 인계 계약으로 취급한다.

---

## 11. 10단계: 모든 위험도에서 실제 Diff 기반 질문 생성

고정된 일반 질문을 직접 제시하는 방식에서 **검증 관점만 고정하고 질문 문장은 실제 Diff에 맞게 생성하는 방식**으로 변경했다.

질문 수:

- LOW: 3개
- MEDIUM: 5개
- HIGH: 5~8개

질문은 실제 파일, 클래스, 메서드, Query, 설정, 테스트를 포함한다.

확인 관점:

- 해결한 문제와 요청→응답 흐름
- 핵심 도메인 규칙의 구현 위치와 계층 선택 이유
- 정상·실패 시 데이터 상태와 트랜잭션·Rollback
- 가장 위험한 경계·동시성·권한·시간·캐시 조건
- 테스트 보장 범위와 미검증 범위
- 대안과 트레이드오프
- 운영 환경의 잔여 위험

역할:

```text
Codex: 최신 실제 Diff를 읽고 질문 생성
Human: 코드와 문서를 보며 직접 답변
Codex: 최신 Head와 답변을 대조해 문항별 PASS/HOLD
ChatGPT: MEDIUM·HIGH 코드 리뷰와 필요 시 독립 검증
Human: 최종 Merge 판단
```

Codex가 질문과 답변을 모두 작성하는 방식은 이해도 검증으로 인정하지 않는다.

## 12. 최종 운영 흐름

```text
1. AI 상세 Draft Issue 작성
2. Human 담당자 착수 검토
3. AI 재검증 1차
4. Human·AI 쟁점 협의
5. 최종 합의 계약 정리
6. Human READY 승인
7. Codex가 권위 있는 최종 계약 추출
8. 작업 브랜치 자동 생성
9. Contract Traceability 작성
10. 구현·테스트·검증·Diff 리뷰
11. 계약별 코드·테스트 증거 연결
12. 전 항목 PASS 확인
13. PR 템플릿 Fidelity 확인
14. 실제 Diff 기반 질문 생성
15. Commit·Push·Draft PR
16. 위험도별 코드 리뷰와 CI
17. Human 직접 답변
18. Codex 답변 검증 PASS/HOLD
19. 필요 시 ChatGPT 독립 검증
20. Human Merge
```

## 13. 개선된 핵심 원칙

```text
AI는 판단 가능한 상세 설계안을 제공한다.
Human은 자신의 언어로 정책과 트레이드오프를 검토한다.
READY는 구현 시작 승인이다.
Codex는 최종 계약만 권위 있는 기준으로 추출한다.
테스트 통과와 계약 전체 충족을 구분한다.
계약은 코드와 테스트 증거에 1:1로 연결한다.
PR 템플릿은 축약하지 않는다.
이해도 질문은 실제 Diff 이후 Codex가 생성한다.
Human은 직접 답변하고 Codex는 최신 코드와 대조한다.
Merge는 Human만 수행한다.
```

프롬프트를 길게 만드는 대신 저장소의 계약과 실행 Gate를 강화한다.

## 14. 포트폴리오 활용 문장

> AI가 생성한 Issue가 빈 템플릿에 가까워 개발자가 설계를 처음부터 작성해야 하는 문제와, 반대로 AI가 Human 검토와 승인까지 대신하는 문제를 모두 경험했습니다. 이를 개선해 AI가 현재 코드와 문서를 근거로 상세 설계안과 대안을 제시하고, 개발자가 자신의 언어로 정책과 트레이드오프를 검토한 뒤 최종 계약을 승인하도록 역할을 분리했습니다. 이후 상세 Issue가 있어도 Codex가 일부 계약과 PR 템플릿을 누락하는 사례를 발견해, 최종 계약을 코드·테스트 증거에 1:1로 연결하는 Contract Traceability Gate와 PR Template Fidelity Gate를 도입했습니다. 또한 구현 후에는 Codex가 실제 Diff 기반 질문을 생성하고 개발자의 답변을 최신 코드와 대조해 PASS/HOLD하는 절차를 추가해, 구현 속도뿐 아니라 계약 준수와 코드 이해까지 검증하도록 개선했습니다.
