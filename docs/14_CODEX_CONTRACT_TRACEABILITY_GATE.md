# Codex Contract Traceability & PR Review Gate

## 1. 목적

이 문서는 `READY` Issue 구현 이후 다음 세 가지를 분리한다.

1. **Contract Traceability Gate**: 최종 구현 계약이 실제 코드와 테스트로 충족됐는지 `PASS | HOLD | N/A`로 추적한다.
2. **Human 이해도 기록**: 작성자가 실제 Diff를 읽고 자신의 말로 답변한다.
3. **PR Review Gate**: ChatGPT가 Human 답변을 포함한 PR 전체를 검증하고, Human이 반영 범위를 결정하며, 수정 후 최신 Head를 다시 검토한다.

`PASS | HOLD | N/A`는 코드·계약 검증에만 사용한다. Codex는 Human 답변을 평가하거나 보완 답변을 작성하지 않는다.

실제 Diff 질문은 `docs/15_DIFF_QUESTION_POLICY.md`, PR 형식은 `.github/PULL_REQUEST_TEMPLATE.md`를 따른다.

## 2. 최종 워크플로우

```text
Codex 구현·Draft PR
→ Human 이해도 작성
→ ChatGPT PR 전체 검증 및 리뷰 댓글
→ Human 반영 범위 결정
→ Codex 승인된 리뷰 수정
→ ChatGPT 최신 Head 재검토
→ Human 최종 Merge
```

### 역할

| 주체 | 책임 |
|---|---|
| Codex | 구현·검증·계약 증거 연결·질문 생성·Draft PR·승인된 리뷰 수정 |
| Human 작성자 | 실제 Diff를 읽고 이해도 답변 작성 |
| ChatGPT | PR 전체 검증·리뷰 댓글·수정 후 최신 Head 재검토 |
| Human 리뷰어 | 리뷰 반영 범위 결정·최종 Merge |

## 3. 기존 방식에서 제거하는 역할

다음 흐름은 사용하지 않는다.

```text
Human 답변
→ Codex가 정답처럼 평가
→ Codex가 보완 답변 작성
→ Human이 AI 문장을 복사
```

제거 이유:

- 이해 확인이 아니라 표현 채점으로 변질된다.
- 구현 주체인 Codex가 자신의 결과와 Human 이해를 동시에 평가하면 검증 독립성이 약해진다.
- Human의 판단과 책임 범위가 불명확해진다.

따라서 Codex는 질문 생성까지만 담당하고, Human 답변을 포함한 전체 PR 검증은 ChatGPT가 담당한다.

## 4. 권위 있는 계약 영역

Codex는 Issue 전체를 읽되 구현 기준은 다음 영역이다.

1. Human·AI 최종 합의
2. 최종 구현 계약
3. 완료 조건
4. 테스트·직접 검증 계획
5. 포함·제외 범위
6. 수정 허용·금지 범위
7. 위험도와 승인 상태

권위 있는 영역끼리 충돌하거나 현재 코드에서 이행할 수 없으면 임의 선택하지 않고 Troubleshooting Gate로 전환한다.

## 5. Contract Traceability Gate

### 5.1 구현 전 계약 추적표

| ID | 최종 계약 | 예상 구현 위치 | 필요한 증거 |
|---|---|---|---|
| C-01 | 구체적인 요구사항 또는 구현 제약 | 클래스·메서드·Query·설정 | 테스트·직접 검증·문서 |

반드시 별도 항목으로 다룬다.

- API·요청·응답 계약
- 도메인 상태와 데이터 원본
- 기간·시간대·정렬·동률·빈 결과 정책
- 트랜잭션·Rollback·권한·동시성 규칙
- Human이 확정한 구현 방식
- 도입 금지 기술과 제외 범위
- 테스트·직접 검증 시나리오

### 5.2 구현 후 증거 연결

| ID | 최종 계약 | 실제 코드 증거 | 테스트·직접 검증 증거 | 상태 |
|---|---|---|---|---|
| C-01 | 구체적인 계약 | `path#symbol` | `TestClass#method` 또는 미검증 사유 | `PASS | HOLD | N/A` |

규칙:

- 모든 계약 항목에 실제 코드 또는 명확한 비코드 근거가 있어야 한다.
- 테스트가 필요한 계약에는 테스트 또는 직접 검증 증거가 있어야 한다.
- `N/A`는 사유를 기록한다.
- 한 항목이라도 `HOLD`이면 최초 Draft PR 생성을 진행하지 않는다.
- 테스트 통과는 추적표 전체 PASS를 대신하지 않는다.

## 6. PR Template Fidelity Gate

Draft PR 생성 직전에 Codex는 최신 `.github/PULL_REQUEST_TEMPLATE.md`를 다시 읽는다.

필수 규칙:

1. 모든 필수 제목을 보존한다.
2. Contract Traceability 표를 실제 증거로 채운다.
3. Codex가 직접 완료한 항목만 체크한다.
4. Human·ChatGPT·CI 단계를 실제 완료 전에 체크하지 않는다.
5. 실제 Diff 기반 질문을 작성하되 Human 답변은 비워 둔다.
6. 질문 작성 전 `docs/15_DIFF_QUESTION_POLICY.md`를 적용한다.
7. Human 답변 평가·보완·중대한 오해 판정 항목을 만들지 않는다.
8. PR 생성 후 본문을 다시 읽는다.

## 7. 실제 Diff 기반 질문

### 7.1 생성 주체와 시점

Codex는 최종 구현·테스트·자체 Diff 리뷰가 끝난 뒤 최신 Diff를 읽고 공식 질문 세트를 한 번 작성한다.

- LOW: 3개
- MEDIUM: 5개
- HIGH: 5~8개

질문 수를 채우기 위해 Issue 핵심과 무관한 세부 구현을 억지로 묻지 않는다.

### 7.2 난이도 원칙

- 신입 Java/Spring 백엔드 기술면접 수준
- 한 문항에 하나의 핵심 개념
- 2~4문장으로 답할 수 있는 범위
- 메서드명·설정 문법보다 코드의 목적과 동작
- 변경 목적·핵심 흐름·테스트 결과 해석을 절반 이상
- LOW·MEDIUM의 세부 구현 질문은 최대 1개
- 위험도가 높아도 질문 자체의 개념 난이도는 높이지 않음

### 7.3 우선 질문 관점

- 변경 전 발생한 문제
- 요청부터 저장 또는 응답까지의 핵심 흐름
- 핵심 도메인 규칙과 데이터 정합성
- 사용한 Spring/JPA/DB 기술의 현재 역할
- 실제 테스트 결과가 의미하는 것
- 테스트가 보장하는 범위와 보장하지 않는 범위
- 다음 Issue에서 달라져야 할 결과

### 7.4 제외할 질문

Issue 핵심과 직접 관련되지 않으면 다음을 묻지 않는다.

- Fixture와 테스트 데이터 삭제 순서
- Foreign Key 정리용 보조 코드
- Gradle DSL과 Test task 내부 동작
- 임시 헬퍼 메서드의 세부 구현
- Spring·Hibernate·Tomcat·HikariCP 내부 동작 암기
- 아직 구현하지 않은 기술과의 심층 비교
- 시니어 운영 장애 분석과 튜닝 수치
- 원인·대안·장단점·운영 위험을 한 번에 묻는 복합 질문

### 7.5 질문 역할 경계

- Codex는 PR 안의 공식 질문만 작성한다.
- Human은 자신의 말로 답변하며 `모르겠다`고 기록할 수 있다.
- Codex는 답변을 평가·보완·대신 작성하지 않는다.
- ChatGPT는 별도 경쟁 질문 세트를 추가하지 않고, 기존 질문의 관련성·난이도·누락을 PR 전체 검증의 일부로 확인한다.
- 부적절한 질문은 Human이 답변하기 전에 교체하는 것이 원칙이다.

## 8. Human 이해도 기록

이 단계의 목적은 시험이 아니라 실제 Diff를 읽고 설명하게 하는 것이다.

- Human 작성자가 질문에 직접 답한다.
- 비담당 리뷰어 참여는 위험도와 Human 결정에 따른다.
- 표현 미숙이나 세부 구현 기억 부족만으로 Merge를 차단하지 않는다.
- Human 답변은 ChatGPT PR 전체 검증의 입력 자료다.
- Codex는 Human 답변 영역을 수정하지 않는다.

## 9. ChatGPT PR 전체 검증

Human 이해도 작성 후 ChatGPT는 최신 Head와 PR 전체를 검증하고 리뷰 댓글을 남긴다.

검증 범위:

- Issue 최종 계약과 실제 Diff 일치
- Contract Traceability 증거의 정확성
- API·ERD·ADR·예외·문서 정합성
- 정상·실패·경계·Rollback·동시성 흐름
- 테스트 의미와 미검증 범위
- Human 답변에서 드러난 핵심 흐름 오해나 누락
- 범위 이탈과 불필요한 변경

ChatGPT는 수정 필요 항목을 근거와 함께 댓글로 남기지만 반영 범위를 결정하거나 Merge하지 않는다.

## 10. Human 반영 범위 결정

Human은 각 리뷰 항목을 다음 중 하나로 결정한다.

- `반영`: 현재 PR에서 수정
- `미반영`: 근거를 남기고 현재 설계 유지
- `후속 Issue`: 현재 범위 밖이므로 별도 추적

Codex는 이 결정이 명시되기 전에 리뷰 수정을 시작하지 않는다.

## 11. Codex 승인된 리뷰 수정 Gate

Codex는 `반영`으로 승인된 항목만 수정한다.

1. 최신 Head와 승인 범위를 다시 읽는다.
2. 승인 항목을 파일·코드·테스트에 연결한다.
3. 승인 범위 안에서 최소 변경한다.
4. 관련 테스트, 전체 검증, Diff 리뷰를 다시 수행한다.
5. Contract Traceability와 검증 증거를 갱신한다.
6. Commit·Push 후 ChatGPT 최신 Head 재검토로 인계한다.

금지:

- 승인되지 않은 리뷰 항목 수정
- Human 답변 문구 수정
- 설계 재결정을 임의 처리
- ChatGPT 재검토·CI·Merge 선행 완료 표시

## 12. ChatGPT 최신 Head 재검토

ChatGPT는 수정 후 최신 Head에서 다음을 확인한다.

- 승인된 리뷰 항목이 의도대로 반영됐는지
- 새로운 회귀나 범위 이탈이 없는지
- 테스트·문서·Contract Traceability가 최신 코드와 일치하는지
- 미해결 리뷰와 잔여 위험이 무엇인지

재검토 결과는 Human 최종 Merge 판단의 근거이며 Merge 승인은 아니다.

## 13. Merge 차단 기준

모든 위험도에서 다음은 Merge 차단이다.

- Contract Traceability `HOLD`
- 최신 CI 실패
- 미해결 설계 충돌
- 승인된 필수 리뷰 미반영
- ChatGPT 최신 Head 재검토 미완료
- 데이터·권한·Rollback·동시성·원본 데이터에 관한 중대한 위험 미해결

단순히 질문이 어렵거나 Human이 세부 구현을 기억하지 못한다는 이유만으로 Merge를 차단하지 않는다. 먼저 질문 정책 위반 여부를 확인한다.

## 14. 금지사항

- Issue 최종 계약보다 최초 제안이나 구현 편의를 우선하지 않는다.
- 테스트 통과만으로 계약 전체 충족을 선언하지 않는다.
- PR 템플릿을 자체 요약 본문으로 대체하지 않는다.
- Human·ChatGPT·CI 단계를 선행 체크하지 않는다.
- Codex가 Human 답변을 평가·보완·대신 작성하지 않는다.
- Codex가 중대한 오해 여부를 최종 판정하지 않는다.
- ChatGPT가 리뷰 반영 범위나 Merge를 결정하지 않는다.
- 승인되지 않은 리뷰를 Codex가 임의로 반영하지 않는다.
- ChatGPT, Codex, 자동화는 Merge하지 않는다. 최종 Merge는 Human만 수행한다.
