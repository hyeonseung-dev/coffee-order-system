# Codex Contract Traceability & Team Understanding Review

## 1. 목적

이 문서는 `READY` Issue를 Codex가 구현할 때 다음 두 가지를 분리해 관리한다.

1. **코드·계약 검증**: 최종 구현 계약이 실제 코드와 테스트로 충족됐는지 `PASS | HOLD | N/A`로 추적한다.
2. **팀 이해·오해 점검**: 작성자와 필요한 팀원이 실제 Diff를 읽고 핵심 변경을 자신의 말로 설명하며, 중요한 오해가 있으면 실제 코드 근거로 보완한다.

코드 품질과 계약 충족은 객관적인 Gate다. 사람의 이해 점검은 AI 정답 시험이 아니다.

실제 Diff 기반 질문의 난이도와 구성은 반드시 `docs/15_DIFF_QUESTION_POLICY.md`를 따른다. 질문 개수와 PR 형식은 `.github/PULL_REQUEST_TEMPLATE.md`를 따르며, 세 문서가 충돌하면 더 구체적인 난이도 정책인 `docs/15_DIFF_QUESTION_POLICY.md`를 우선한다.

## 2. 이전 PR에서 확인한 실패

### 2.1 PR 템플릿 축약

Codex가 최신 PR 템플릿을 그대로 사용하지 않으면 계약 추적, 리뷰 상태, 이해 점검 영역이 누락될 수 있다.

### 2.2 확정 계약 일부 누락

Issue에 확정된 시간대, Fixture 방식, 경계 테스트처럼 테스트가 통과해도 자동으로 검증되지 않는 계약이 있다. 따라서 계약별 코드·테스트 증거 연결이 필요하다.

### 2.3 이해도 Gate의 부작용

Human 답변을 AI가 정답처럼 채점하면 다음 흐름이 생길 수 있다.

```text
Human이 자기 언어로 답변
→ AI가 원하는 표현과 다르다고 HOLD
→ AI 문장을 복사
→ 형식상 통과
```

이 방식은 이해가 아니라 정답 복사를 강화한다. 따라서 이해 점검은 실제 코드 대조와 보완 설명 중심으로 운영하고, Merge 차단은 위험도와 중대한 오해 여부에 따라 Human이 결정한다.

### 2.4 질문 난이도 과잉

실제 Diff에 존재한다는 이유만으로 테스트 보조 코드, Gradle DSL, Fixture 삭제 순서, 프레임워크 내부 동작을 질문하면 Issue의 핵심 학습 목표가 흐려진다.

위험도가 높아도 질문 자체의 개념 난이도를 높이지 않는다. 위험도는 질문 수와 확인할 실패 시나리오의 범위만 늘린다.

## 3. 권위 있는 계약 영역

Codex는 Issue 전체를 읽되 구현 기준은 다음 영역이다.

1. Human·AI 최종 합의
2. 최종 구현 계약
3. 완료 조건
4. 테스트·직접 검증 계획
5. 포함·제외 범위
6. 수정 허용·금지 범위
7. 위험도와 승인 상태

권위 있는 영역끼리 충돌하거나 현재 코드에서 이행할 수 없으면 임의 선택하지 않고 Troubleshooting Gate로 전환한다.

## 4. Contract Traceability Gate

### 4.1 구현 전 계약 추적표

Codex는 최종 계약을 원자 항목으로 나눈다.

| ID | 최종 계약 | 예상 구현 위치 | 필요한 증거 |
|---|---|---|---|
| C-01 | 구체적인 요구사항 또는 구현 제약 | 클래스·메서드·Query·설정 | 테스트·직접 검증·문서 |

반드시 별도 항목으로 다루는 것:

- API·요청·응답 계약
- 도메인 상태와 데이터 원본
- 기간·시간대·정렬·동률·빈 결과 정책
- 트랜잭션·Rollback·권한·동시성 규칙
- Human이 확정한 구현 방식
- 도입 금지 기술과 제외 범위
- 테스트·직접 검증 시나리오

### 4.2 구현 후 증거 연결

| ID | 최종 계약 | 실제 코드 증거 | 테스트·직접 검증 증거 | 상태 |
|---|---|---|---|---|
| C-01 | 구체적인 계약 | `path#symbol` | `TestClass#method` 또는 미검증 사유 | `PASS | HOLD | N/A` |

규칙:

- 모든 계약 항목에 실제 코드 또는 명확한 비코드 근거가 있어야 한다.
- 테스트가 필요한 계약에는 테스트 또는 직접 검증 증거가 있어야 한다.
- `N/A`는 사유를 기록한다.
- 한 항목이라도 `HOLD`이면 완료 보고, Commit, Draft PR 생성을 진행하지 않는다.
- 테스트 통과는 추적표 전체 PASS를 대신하지 않는다.
- `PASS | HOLD`는 코드·계약 검증에만 사용한다.

## 5. PR Template Fidelity Gate

Draft PR 생성 직전에 Codex는 베이스 브랜치의 최신 `.github/PULL_REQUEST_TEMPLATE.md`를 다시 읽는다.

필수 규칙:

1. 모든 `##`·`###` 제목을 보존한다.
2. 빈 섹션을 자체 요약 문장으로 대체하거나 삭제하지 않는다.
3. Contract Traceability 표를 실제 증거로 채운다.
4. Codex가 직접 완료한 항목만 체크한다.
5. Human·ChatGPT·팀원·독립 리뷰·CI·이해 점검은 실제 완료 전에 체크하지 않는다.
6. 실제 Diff 기반 질문을 작성하되 Human·팀원 답변은 비워 둔다.
7. 질문을 작성하기 전에 `docs/15_DIFF_QUESTION_POLICY.md`의 자체 검수 체크리스트를 적용한다.
8. PR 생성 후 본문을 다시 읽어 템플릿 누락, 선행 체크, 질문 난이도 위반을 확인한다.

## 6. 실제 Diff 기반 질문

### 6.1 생성 주체와 시점

Codex는 최종 구현·테스트·자체 Diff 리뷰가 끝난 뒤 최신 Diff를 읽고 공식 질문 세트를 한 번만 작성한다.

- LOW: 3개
- MEDIUM: 5개
- HIGH: 5~8개

질문 수를 채우기 위해 Issue 핵심과 무관한 세부 구현을 억지로 묻지 않는다.

### 6.2 난이도 권위 문서

질문 생성과 검수는 `docs/15_DIFF_QUESTION_POLICY.md`를 반드시 따른다.

핵심 원칙:

- 신입 Java/Spring 백엔드 기술면접 수준을 유지한다.
- 한 문항에는 하나의 핵심 개념만 묻는다.
- 신입 개발자가 2~4문장으로 답할 수 있어야 한다.
- 메서드명과 설정 문법 자체보다 코드의 목적과 동작을 묻는다.
- 변경 목적·핵심 흐름·테스트 결과 해석을 전체 질문의 절반 이상으로 구성한다.
- LOW·MEDIUM의 세부 구현 질문은 최대 1개다.
- 위험도가 높다는 이유로 질문 난이도를 높이지 않는다.

### 6.3 우선 질문 관점

다음 관점을 실제 코드와 테스트 결과에 연결한다.

- 변경 전 발생한 문제
- 요청부터 저장 또는 응답까지의 핵심 흐름
- 핵심 도메인 규칙과 데이터 정합성
- 사용한 Spring/JPA/DB 기술이 현재 코드에서 맡은 역할
- 실제 테스트 결과가 의미하는 것
- 테스트가 보장하는 범위와 보장하지 않는 범위
- 다음 Issue에서 달라져야 할 결과

### 6.4 원칙적으로 제외할 질문

다음은 Issue 핵심 학습 목표와 직접 관련되지 않으면 질문하지 않는다.

- Fixture와 테스트 데이터 삭제 순서
- Foreign Key 정리용 보조 코드
- Gradle DSL과 Test task 내부 동작
- 임시 헬퍼 메서드의 세부 구현
- Spring·Hibernate·Tomcat·HikariCP 내부 동작 암기
- 아직 구현하지 않은 기술과의 심층 비교
- 시니어 운영 장애 분석과 튜닝 수치
- 한 문항에서 원인·대안·장단점·운영 위험을 모두 묻는 복합 질문

### 6.5 생성 후 필수 검수

PR에 기록하기 전에 다음을 확인한다.

- [ ] 모든 질문이 해당 Issue의 작업 목적과 직접 관련된다.
- [ ] 변경 목적·핵심 흐름·결과 해석 질문이 전체의 절반 이상이다.
- [ ] 한 질문에 핵심 개념이 하나만 있다.
- [ ] 각 질문은 2~4문장으로 답할 수 있다.
- [ ] 프레임워크 내부 구현 암기를 요구하지 않는다.
- [ ] Fixture 삭제, 정리 순서, Gradle DSL을 핵심 질문으로 사용하지 않았다.
- [ ] LOW·MEDIUM의 세부 구현 질문은 최대 1개다.
- [ ] 위험도를 이유로 난이도를 높이지 않았다.

하나라도 충족하지 못하면 질문을 PR에 기록하기 전에 교체한다.

### 6.6 AI 질문 역할 경계

- Codex가 PR 안의 공식 질문 세트를 작성한다.
- ChatGPT는 별도의 경쟁 질문 세트를 추가하지 않는다.
- ChatGPT와 리뷰어는 기존 질문의 Issue 관련성, 난이도, 중대한 누락을 검수한다.
- 부적절한 질문은 Human이 답변하기 전에 교체한다.
- Codex와 ChatGPT는 Human 답변을 대신 작성하지 않는다.

## 7. 팀 이해·오해 점검

### 7.1 목적

이 단계는 시험이 아니다.

- 실제 Diff를 한 번 읽게 한다.
- 담당자만 아는 코드를 줄인다.
- 모르는 부분을 숨기지 않게 한다.
- 중대한 오해를 Merge 전에 발견한다.

Human은 `모르겠다`고 답할 수 있다. AI는 정답 재제출을 요구하지 않고 실제 코드 근거와 보완 설명을 제공한다.

### 7.2 위험도별 참여 기준

#### LOW

- 작성자가 실제 Diff 질문에 답한다.
- 비담당 리뷰어 1명이 확인한다.
- 이해 점검 자체는 Merge 차단 조건이 아니다.
- 중대한 오해가 있을 때만 Merge를 보류한다.

#### MEDIUM

- 작성자가 실제 Diff 질문에 답한다.
- 비담당 리뷰어 최소 1명이 변경 목적, 핵심 흐름, 실패 위험을 자기 언어로 요약한다.
- 중대한 오해가 있을 때만 Merge를 보류한다.

#### HIGH

- 작성자가 질문에 답한다.
- 비담당 리뷰어 최소 2명이 이해 요약을 남긴다.
- 팀 공통 영역에 영향이 크면 팀원 전원이 확인한다.
- 필수 참여자의 점검과 보완 설명 확인이 끝날 때까지 Merge를 보류한다.
- 중대한 오해가 남아 있으면 계속 보류한다.

### 7.3 역할

```text
Codex
→ 실제 코드와 답변의 일치·불일치 근거 제시

ChatGPT
→ 공식 질문의 난이도 검수, 오해와 미검증 범위 설명

Human 리뷰어
→ 중대한 오해인지 최종 판단

Human
→ 최종 Merge 결정
```

AI는 사람의 이해도를 최종 채점하거나 Merge를 승인하지 않는다.

### 7.4 상태 표현

이해 점검에는 `PASS | HOLD` 대신 다음 상태를 사용한다.

- `확인 완료`: 설명이 실제 코드 흐름과 대체로 일치한다.
- `보완 설명 필요`: 일부 오해·누락이 있어 AI 설명이 필요하다.
- `보완 설명 완료`: AI 설명을 확인했고 중대한 오해가 남아 있지 않다.
- `차단 필요`: Merge 후 사고로 이어질 수 있는 중대한 오해가 남아 있다.

### 7.5 중대한 오해

다음처럼 데이터·보안·운영 사고로 이어질 수 있는 오해다.

- 금액·포인트·재고 정합성을 반대로 이해
- 인증·인가 보호 대상을 잘못 이해
- 트랜잭션과 Rollback 범위를 잘못 이해
- 동시성 락이 무엇을 보호하는지 잘못 이해
- 캐시를 원본 데이터로 오인
- 데이터 삭제·Migration 영향을 잘못 이해
- 메시지 중복·유실 가능성을 잘못 이해
- 테스트하지 않은 범위를 검증했다고 판단

단순 용어 부족, 표현 미숙, 세부 구현 기억 부족은 중대한 오해가 아니다.

### 7.6 보완 절차

```text
Human·팀원 답변
→ AI가 실제 코드와 다른 부분을 설명
→ Human·팀원이 설명 확인 기록
→ Human 리뷰어가 중대한 오해 잔존 여부 판단
```

모범 답안 재작성이나 AI 문장 복사를 요구하지 않는다.

## 8. 최종 실행 흐름

```text
READY Issue
→ 최종 계약 추출
→ Contract Traceability 작성
→ 구현·테스트·실제 Diff 리뷰
→ 계약별 코드·테스트 증거 연결
→ Contract Traceability 전 항목 PASS
→ 최신 PR 템플릿 확인
→ docs/15 기준 실제 Diff 질문 생성·난이도 검수
→ Commit·Push·Draft PR
→ 코드 리뷰·수정·최신 CI
→ 작성자와 필요한 팀원 답변
→ AI 코드 대조·보완 설명
→ Human 중대한 오해 여부 판단
→ 위험도별 이해 점검 완료
→ Human Merge
```

## 9. Merge 차단 기준

### 모든 위험도

다음은 Merge 차단이다.

- Contract Traceability `HOLD`
- 최신 CI 실패
- 미해결 설계 충돌
- 데이터·권한·Rollback·동시성 보호 대상에 관한 중대한 오해가 남아 있음

단순히 질문이 어렵거나 작성자가 세부 구현을 기억하지 못한다는 이유만으로 Merge를 차단하지 않는다. 먼저 질문 난이도가 `docs/15_DIFF_QUESTION_POLICY.md`를 위반했는지 확인한다.

### HIGH 추가 조건

다음이 완료될 때까지 Merge를 보류한다.

- 작성자 답변
- 비담당 리뷰어 최소 2명 확인
- 필요한 독립 AI 리뷰
- 강화된 Human 리뷰
- 위험에 맞는 테스트
- AI 보완 설명 확인

## 10. 금지사항

- Issue 최종 계약보다 최초 제안이나 구현 편의를 우선하지 않는다.
- 테스트 통과만으로 계약 전체 충족을 선언하지 않는다.
- PR 템플릿을 자체 요약 본문으로 대체하지 않는다.
- Human·ChatGPT·팀원·CI·이해 점검을 선행 체크하지 않는다.
- Codex나 ChatGPT가 Human 답변을 대신 작성하지 않는다.
- 질문 개수를 채우기 위해 Issue와 무관한 세부 구현을 묻지 않는다.
- 위험도가 높다는 이유로 질문 자체의 개념 난이도를 높이지 않는다.
