# Codex Contract Traceability & Diff-based Understanding Gate

## 1. 목적

이 문서는 `READY` Issue를 한 문장으로 Codex에 전달해도 다음이 누락되지 않도록 실행 단계를 강제한다.

- Human이 확정한 최종 구현 계약
- 완료 조건과 테스트·직접 검증 기준
- 제외 범위와 구현 방식 제약
- 베이스 브랜치의 Pull Request 템플릿
- 실제 Diff에 맞춘 구현 후 이해도 질문
- Human 답변과 실제 코드·테스트의 일치 검증

프롬프트를 길게 반복하지 않는다. 상세 요구사항은 Issue와 저장소 문서에 한 번만 기록하고, Codex는 이 문서에 따라 추적·검증한다.

## 2. PR #32에서 확인한 실패

### 2.1 PR 템플릿 축약

Codex는 `.github/PULL_REQUEST_TEMPLATE.md` 전체를 사용하지 않고 자체 요약 본문을 생성했다. 그 결과 다음이 사라지거나 사실과 다르게 표시됐다.

- 구현 후 이해도 검증 질문 영역
- 계약별 구현·테스트 증거
- Human과 ChatGPT가 아직 수행하지 않은 리뷰 체크 상태

원인은 단순한 모델 실수만이 아니다. 기존 Skill은 PR 본문에 위험도와 검증 증거를 기록하라고만 했고, **베이스 브랜치의 최신 템플릿을 읽고 모든 제목을 보존하라**는 하드 규칙이 없었다.

### 2.2 확정 계약 일부 누락

Issue #9에는 `Asia/Seoul` Clock, SQL 또는 `JdbcTemplate` 기반 시간 Fixture, TOP 3 초과·3개 미만·0개 시나리오가 확정돼 있었다. 그러나 구현과 테스트 일부가 계약과 달랐다.

테스트·빌드가 통과했는데도 누락된 이유는 다음과 같다.

1. Issue 본문에 최초 제안, Human 원문, 재검증, 최종 합의가 함께 있어 Codex가 최종 계약만 구조적으로 추출하지 않았다.
2. 자체 리뷰가 `Issue 완료 조건 충족`이라는 단일 체크에 머물렀고 요구사항별 코드·테스트 대응표가 없었다.
3. 테스트는 작성된 코드와 작성된 테스트의 일관성만 검증했다. 작성하지 않은 경계 테스트는 실패할 수 없다.
4. Service 테스트는 KST Clock Mock을 사용했지만 실제 `ClockConfig` Bean은 검증하지 않아 설정 불일치를 잡지 못했다.
5. Reflection Fixture도 조회 결과는 맞게 만들 수 있어, 동작 검증만으로는 Human이 정한 구현 방식 제약 위반을 발견하지 못했다.

따라서 해결책은 Issue를 더 길게 쓰는 것이 아니라 **최종 계약 추출과 증거 추적을 실행 Gate로 만드는 것**이다.

## 3. 권위 있는 계약 영역

Codex는 Issue 전체를 읽되 구현 기준은 다음 영역만 권위 있는 계약으로 취급한다.

1. Human·AI 최종 합의
2. 최종 구현 계약
3. 완료 조건
4. 테스트·직접 검증 계획
5. 포함·제외 범위
6. 수정 허용·금지 범위
7. 위험도와 승인 상태

AI 최초안, Human 최초 이해, 재검증 기록은 배경과 학습 기록이다. 최종 계약과 충돌하면 최종 계약을 따른다.

권위 있는 영역끼리 충돌하거나 현재 코드상 이행할 수 없으면 임의 선택하지 않고 Troubleshooting Gate로 전환한다.

## 4. Contract Traceability Gate

### 4.1 구현 전 계약 추적표

Codex는 구현 전에 최종 계약을 원자적인 항목으로 나누고 내부 작업 목록을 만든다.

| ID | 최종 계약 | 예상 구현 위치 | 필요한 증거 |
|---|---|---|---|
| C-01 | 구체적인 요구사항 | 클래스·메서드·Query·설정 | 테스트·직접 검증·문서 |

계약 항목에는 결과뿐 아니라 Human이 확정한 구현 제약도 포함한다.

예:

- `Asia/Seoul` 고정
- JOIN 조건을 `ON` 절에 배치
- SQL 또는 `JdbcTemplate`로 Fixture 생성
- QueryDSL 도입 금지
- 외부 Query Parameter 노출 금지

### 4.2 구현 후 증거 연결

Commit 전에 표를 실제 증거로 갱신한다.

| ID | 최종 계약 | 실제 코드 증거 | 테스트·직접 검증 증거 | 상태 |
|---|---|---|---|---|
| C-01 | 구체적인 요구사항 | `path#symbol` | `TestClass#method` 또는 미검증 사유 | `PASS | HOLD | N/A` |

규칙:

- 모든 계약 항목에 실제 코드 또는 명확한 비코드 근거가 있어야 한다.
- 테스트가 필요한 계약에는 테스트 또는 직접 검증 증거가 있어야 한다.
- `N/A`는 사유를 기록해야 한다.
- 한 항목이라도 `HOLD`이면 완료 보고, Commit, Draft PR 생성을 진행하지 않는다.
- 구현 방식 제약은 결과가 같더라도 다른 방식으로 대체하지 않는다.
- 테스트 통과는 추적표 전체 PASS를 대신하지 않는다.

## 5. PR Template Fidelity Gate

Draft PR 생성 직전에 Codex는 **베이스 브랜치의 최신** `.github/PULL_REQUEST_TEMPLATE.md`를 다시 읽는다.

필수 규칙:

1. 템플릿의 모든 `##`·`###` 제목을 보존한다.
2. 빈 섹션을 자체 요약 문장으로 대체하거나 삭제하지 않는다.
3. Contract Traceability 표를 실제 코드와 테스트 증거로 채운다.
4. Codex가 직접 완료한 항목만 체크한다.
5. Human 리뷰, ChatGPT 리뷰, 독립 리뷰, CI, 이해도 검증은 실제 완료 전 체크하지 않는다.
6. PR 생성 후 본문을 다시 읽어 템플릿 제목 누락과 선행 체크를 확인한다.
7. 템플릿을 보존할 수 없는 PR 생성 수단이라면 Push까지 완료하고 PR 생성 단계만 `BLOCKED`로 보고한다.

`gh` CLI 부재는 Commit·Push 중단 사유가 아니다.

- Commit과 Push: 표준 `git` 명령
- Draft PR: 사용 가능한 GitHub 연동, API 또는 `gh`
- PR 생성 수단이 없을 때만 Push 후 compare 정보와 PR 본문을 제공

## 6. 실제 Diff 기반 이해도 질문

### 6.1 생성 주체와 시점

Codex는 최종 구현·테스트·자체 Diff 리뷰가 끝난 뒤, 최신 Diff를 읽고 PR 본문에 질문을 작성한다.

- Codex는 질문만 작성한다.
- Human 답변을 대신 작성하지 않는다.
- 고정된 일반 질문을 그대로 복사하지 않는다.
- 질문마다 실제 파일, 클래스, 메서드, Query, 설정 또는 테스트를 포함한다.

질문 수:

- `LOW`: 3개
- `MEDIUM`: 5개
- `HIGH`: 5~8개

### 6.2 반드시 확인할 관점

질문의 문장은 실제 Diff에 맞게 바꾸되 다음 관점을 위험도에 맞게 포함한다.

- 이 변경이 해결하는 실제 문제와 요청→응답 흐름
- 핵심 도메인 규칙의 구현 위치와 해당 계층에 둔 이유
- 정상·실패 시 데이터 상태와 트랜잭션·Rollback 경계
- 가장 위험한 경계·동시성·권한·캐시·시간 조건
- 테스트가 보장하는 범위와 보장하지 못하는 범위
- 선택한 구현과 제외한 대안의 트레이드오프
- 운영 환경에서 남는 위험

### 6.3 좋은 질문 예시 — PR #32 기준

아래 문장을 고정 템플릿으로 복사하지 않는다. 실제 Diff 기반 질문의 구체성 수준을 보여주는 예시다.

1. `ClockConfig.clock()`과 `MenuService.findPopularMenus()`를 함께 볼 때, 서버 기본 시간대가 아니라 `Asia/Seoul` Clock을 고정해야 하는 이유와 UTC 서버에서 자정 부근에 발생할 수 있는 오류를 설명하라.
2. `MenuRepository.findPopularMenus()`에서 주문 상태와 기간 조건을 `WHERE`가 아니라 `LEFT JOIN ... ON`에 둔 이유를 주문 0건 ACTIVE 메뉴 결과와 연결해 설명하라.
3. 같은 Query에서 `COUNT(o.id)`와 `COUNT(*)`의 결과가 어떻게 달라질 수 있으며, `orderCount DESC, menuId ASC`가 왜 결정적인 TOP 3 결과를 만드는지 설명하라.
4. `PageRequest.of(0, 3)`이 실제로 4개 이상의 후보에서 3개만 반환한다는 것을 어느 테스트가 증명하는지, 증명하지 못한다면 어떤 테스트가 필요한지 설명하라.
5. H2 Repository 테스트가 확인하는 것과 실제 MySQL 실행계획·시간대·데이터 규모에서 보장하지 못하는 것을 구분하라.

### 6.4 Human 답변 검증

Human이 PR 본문에 답변한 뒤 Codex는 최신 Head의 Diff와 테스트를 다시 읽고 답변을 검증한다.

검증 결과:

- `PASS`: 답변이 실제 코드·테스트와 일치하고 핵심 선택과 한계를 설명함
- `HOLD`: 코드와 다른 설명, 과장된 테스트 보장, 핵심 경계·실패 상태·선택 이유 누락

Codex는 다음을 PR에 기록한다.

- 문항별 `PASS | HOLD`
- 실제 코드와 일치하는 근거
- 수정이 필요한 오해
- 미검증·잔여 위험
- 전체 결과 `PASS | HOLD`

MEDIUM·HIGH PR의 코드 리뷰와 최종 Merge 판단은 별도 Human/ChatGPT 검토를 유지한다. Codex의 이해도 검증 PASS는 Merge 권한이 아니다.

## 7. 최종 실행 흐름

```text
READY Issue
→ 권위 있는 최종 계약 추출
→ 구현 전 Contract Traceability 표 작성
→ 작업 브랜치 자동 생성
→ 구현·테스트·검증
→ 실제 Diff 자체 리뷰
→ 계약별 코드·테스트 증거 연결
→ 전 항목 PASS 확인
→ 베이스 브랜치 PR 템플릿 재확인
→ 실제 Diff 기반 질문 생성
→ Commit·Push·Draft PR
→ 코드 리뷰·수정·최신 CI
→ Human 직접 답변
→ Codex 답변 검증 PASS/HOLD
→ 필요한 독립 리뷰
→ Human Merge
```

## 8. 금지사항

- Issue 최종 계약보다 최초 제안이나 구현 편의를 우선하지 않는다.
- 테스트가 통과했다는 이유만으로 계약 전체 충족을 선언하지 않는다.
- PR 템플릿을 자체 요약 본문으로 대체하지 않는다.
- Human·ChatGPT·CI·이해도 Gate를 선행 체크하지 않는다.
- 일반적인 질문만 남기고 실제 Diff 기반 질문 생성을 생략하지 않는다.
- Codex가 Human 답변을 작성하거나 Merge하지 않는다.
