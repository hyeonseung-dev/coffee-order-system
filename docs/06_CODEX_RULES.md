# Codex Rules

Codex는 이 프로젝트에서 **Issue 단위 코드 구현과 로컬 검증**을 담당한다.

요구사항 정리, 설계 문서, README, API 명세, 작업 규칙, 리뷰 로그 같은 Markdown 문서는 기본적으로 ChatGPT가 작성·수정한다.

## 작업 범위

- Issue 범위 안에서만 작업한다.
- Issue에 없는 기능을 추가하지 않는다.
- 불필요한 리팩터링을 하지 않는다.
- 패키지 구조를 임의로 변경하지 않는다.
- 승인된 Issue와 작업 범위 안에서만 코드와 테스트 파일을 수정한다.
- Markdown 문서는 Issue에서 Codex 수정 대상으로 명시한 경우에만 수정한다.
- 구현 결과에 따라 문서 보완이 필요하면 PR 본문에 `ChatGPT 문서 보완 필요` 항목으로 남긴다.

하네스 실행 절차와 Attempt·Record·Enforce 규칙은 `.codex/skills/coffee-order-issue-loop/SKILL.md`를 따른다.

## 작업 전 보고

작업 전 반드시 다음을 확인하고 보고한다.

```bash
git branch --show-current
git status
git log --oneline -5
```

보고 내용:

- 현재 브랜치
- 작업 트리 상태
- 수정 예정 코드·테스트 파일 목록
- 각 파일을 수정하는 이유
- 테스트 계획
- Issue 범위 밖 변경 없음 여부
- 문서 보완이 예상되면 그 내용만 별도 표시

## 보호 파일 판단

보호 여부는 Issue의 수정 허용·금지 범위와 Preflight 판단으로 결정한다. 명시적으로 허용되지 않은 공통 설정 파일과 문서는 수정하지 않는다.

- `build.gradle`
- `settings.gradle`
- `application.yml`
- `application.properties`
- `README.md`
- `AGENTS.md`
- `docs/*.md`
- `.github/**/*.md`

보호 파일 변경이 실제 구현에 필수라면 임의 수정하지 않고 이유와 최소 변경안을 보고한 뒤 Human 승인을 기다린다.

## JavaDoc 및 주석 작성 원칙

Codex는 새로운 클래스를 생성하거나 기존 코드를 수정할 때 학습과 유지보수를 돕는 수준의 JavaDoc과 주석을 함께 검토한다.

- 모든 주요 클래스에는 클래스 책임, 소속 계층, 협력 대상, 실행 조건이나 제약을 설명하는 JavaDoc을 작성한다.
- Controller 메서드에는 어떤 요청을 받아 어떤 Service 흐름을 실행하는지 작성한다.
- Service의 비즈니스 메서드에는 처리 순서, 핵심 규칙, 실패 조건을 작성한다.
- Repository의 파생 쿼리나 커스텀 조회 메서드에는 조회 목적과 조건을 작성한다.
- Entity의 생성 메서드와 상태 변경 메서드에는 생성 상태, 변경 필드, 검증 규칙, setter 대신 메서드를 사용하는 이유를 작성한다.
- DTO에는 어떤 요청 또는 응답을 표현하는지와 주요 필드 의미를 작성한다.
- 예외 처리 메서드에는 어떤 계층에서 발생한 예외인지와 별도 처리 이유를 작성한다.
- 트랜잭션, 동시성, 락, 캐시, 이벤트 처리에는 선택 이유와 보호 대상 데이터를 작성한다.
- 코드만으로 이해하기 어려운 Spring/JPA 동작에는 짧은 인라인 주석을 작성한다.
- 코드 내용을 한국어로 그대로 번역하는 주석, 현재 구현과 맞지 않는 미래 설명, 과장된 일반론, 의미 없는 짧은 주석은 작성하지 않는다.
- getter, JPA 기본 생성자, 단순 위임 메서드처럼 코드 자체로 명확한 부분에는 기계적으로 JavaDoc을 붙이지 않는다.

JavaDoc은 코드의 일부이므로 Codex가 작성할 수 있지만, 프로젝트 설명용 Markdown 문서와 혼동하지 않는다.

## 작업 중단 조건

- 현재 브랜치가 지정 브랜치와 다를 때
- Issue 범위 밖 파일 수정이 필요할 때
- 보호 파일 수정이 필요한데 승인이 없을 때
- 테스트 실패 원인이 현재 Issue와 무관할 때
- 기존 구조를 크게 바꿔야 해결 가능할 때
- 요구사항이 모호할 때
- 데이터 삭제, 마이그레이션, 외부 비용, 보안 정책 변경이 필요할 때

Markdown 문서 보완이 필요한 것만으로는 코드 구현을 BLOCKED 처리하지 않는다. 코드 구현과 검증을 완료하고 PR 본문에 ChatGPT 보완 항목으로 남긴다.

## 구현 후 검증

구현 후 반드시 다음을 실행하고 결과를 확인한다.

```bash
git status
git diff --stat
git diff
./gradlew test
./gradlew build
git diff --check
```

전체 빌드가 테스트를 포함하고 같은 결과를 반복하는 경우 하네스 규칙에 따라 중복 실행을 줄일 수 있다.

## Commit·Push·PR

검증이 끝나면 다음을 수행한다.

1. Issue 범위의 코드와 테스트만 Commit한다.
2. 작업 브랜치를 Push한다.
3. PR을 생성한다.
4. PR 본문에 다음을 기록한다.
   - `Closes #<Issue 번호>`
   - 구현 내용
   - 테스트와 빌드 결과
   - 미검증 항목
   - 범위 이탈 여부
   - ChatGPT가 보완해야 할 Markdown 문서
5. PR 번호를 Human에게 전달한다.

PR 생성 후에는 전체 결과를 대화에 다시 복사하지 않는다. Human이 ChatGPT에 PR 번호를 전달하면 ChatGPT가 GitHub에서 직접 검토한다.

## 로컬 BLOCKED 보고

Push 또는 PR 생성 전에 중단된 경우에만 다음을 보고한다.

- 실패한 명령
- 핵심 오류 메시지
- 현재 브랜치와 작업 트리 상태
- 생성된 Commit 유무
- 수행하지 못한 검증
- 필요한 Human 조치

GitHub Actions에서 발생한 실패는 PR 번호를 전달하고, ChatGPT가 원격 로그를 확인하도록 한다.

## 구현 후 보고

최종 보고 내용:

- PR 번호와 URL
- 변경 코드·테스트 파일 요약
- 테스트·빌드 결과
- 미검증 항목
- ChatGPT 문서 보완 필요 여부
- Issue 범위 이탈 여부

Merge는 Human 승인 후 수행한다.
