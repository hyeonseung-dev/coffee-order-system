# Codex Rules

Codex는 이 프로젝트에서 승인된 READY Issue를 기준으로 코드 구현, 테스트, 로컬 검증, Commit·Push·PR 생성을 담당한다.

요구사항, 설계 문서, README, API 명세, ADR, 워크플로우, 리뷰 로그 등 Markdown 문서는 기본적으로 ChatGPT가 작성·수정한다.

## 1. 구현 시작 조건

다음 조건이 모두 충족돼야 구현한다.

- 대상 GitHub Issue가 존재함
- Issue 상태가 `READY`임
- 목적, 완료 조건, 제외 범위가 명확함
- 수정 허용·금지 범위가 명확함
- 위험도가 `LOW`, `MEDIUM`, `HIGH` 중 하나로 결정됨
- 테스트 계획과 검증 방법이 있음
- 예상 트러블슈팅과 Human 공유 조건이 있음
- 필요한 ADR과 Human 승인이 완료됨

조건이 부족하면 임의로 보충하거나 구현하지 않는다.

## 2. 작업 범위

- Issue 범위 안에서만 작업한다.
- Issue에 없는 기능을 추가하지 않는다.
- 불필요한 리팩터링을 하지 않는다.
- 패키지와 계층 구조를 임의로 변경하지 않는다.
- 기존 API, DTO, 예외, DB 계약을 임의로 변경하지 않는다.
- 승인된 코드·테스트 파일만 수정한다.
- Markdown 문서는 Issue에서 Codex 수정 대상으로 명시한 경우에만 수정한다.
- 구현 결과에 따라 문서 보완이 필요하면 PR 본문에 `ChatGPT 문서 보완 필요`로 남긴다.

## 3. 위험도별 행동

### LOW

기존 패턴을 따르는 명확한 작업이다.

예시:

- 단순 조회
- DTO 또는 Mapper
- Validation
- 테스트 보완
- 작은 버그 수정

Codex는 별도 계획 승인 없이 구현, 테스트, Commit, Push, PR 생성까지 진행한다.

### MEDIUM

설계 선택이나 공통 구조 영향이 있는 작업이다.

예시:

- API 계약
- Entity 관계
- 트랜잭션 범위
- Repository 쿼리와 인덱스
- 캐시
- 이벤트 처리

파일 수정 전에 짧은 계획, 영향 범위, 테스트 기준을 보고하고 Human 승인을 기다린다.

### HIGH

데이터 손실, 보안, 정합성, 운영 장애 가능성이 큰 작업이다.

예시:

- 인증·인가
- 금액·포인트·결제·환불
- 동시성 제어
- DB Migration
- 메시지 전달 보장
- 운영 인프라와 배포 설정

필요한 ADR과 Human 승인 상태를 확인한 뒤에만 구현한다. 새로운 설계 선택이 발견되면 즉시 중단한다.

위험도와 별도 실행 모드를 동시에 사용하지 않는다.

## 4. 작업 전 확인

```bash
git branch --show-current
git status
git log --oneline -5
```

확인 내용:

- 현재 브랜치와 대상 Issue 일치 여부
- 기존 미커밋 변경과 충돌 여부
- 수정 예정 코드·테스트 파일
- 보호 파일 포함 여부
- 관련 테스트와 실행 명령
- Issue 범위 밖 변경 필요 여부

LOW는 간단히 확인 후 진행한다. MEDIUM과 HIGH는 계획·승인 조건도 함께 확인한다.

## 5. 보호 파일

Issue에서 명시적으로 허용하지 않은 다음 파일은 수정하지 않는다.

- `build.gradle`
- `settings.gradle`
- `application.yml`
- `application.properties`
- `README.md`
- `AGENTS.md`
- `docs/*.md`
- `.github/**/*.md`
- `.codex/**/*.md`

보호 파일 변경이 실제 구현에 필수라면 수정 이유, 최소 변경안, 영향, 검증 방법을 보고하고 Human 승인을 기다린다.

## 6. 구현 원칙

- 가장 작은 변경 단위부터 구현한다.
- Controller, Service, Repository, Domain 책임을 유지한다.
- 비즈니스 규칙을 테스트로 표현한다.
- 정상 흐름뿐 아니라 실패와 롤백 흐름을 검증한다.
- 테스트를 통과시키기 위해 요구사항을 왜곡하지 않는다.
- 실패한 테스트를 삭제하거나 비활성화하지 않는다.
- 새로운 의존성이나 인프라를 임의로 추가하지 않는다.
- 현재 Issue와 무관한 문제는 별도 보고한다.
- 실행하지 않은 명령은 성공으로 기록하지 않는다.

## 7. JavaDoc과 주석

JavaDoc은 코드의 일부이므로 Codex가 작성할 수 있다.

- 주요 클래스에는 책임과 협력 대상을 설명한다.
- Service 비즈니스 메서드에는 핵심 규칙과 실패 조건을 설명한다.
- 트랜잭션, 동시성, 락, 캐시, 이벤트에는 선택 이유와 보호 대상을 설명한다.
- 코드 내용을 그대로 번역하는 주석은 작성하지 않는다.
- 단순 getter, 기본 생성자, 명확한 위임 메서드에 기계적으로 작성하지 않는다.

## 8. 일반 디버깅과 Troubleshooting Gate

### 자체 해결 가능한 일반 오류

Issue 범위 안에서 해결하고 PR에 필요한 경우만 짧게 기록한다.

- 컴파일 오류
- import 누락
- 단순 테스트 데이터 오류
- 명확한 Mock 설정 오류
- 코드 스타일과 정적 검사 오류
- 기존 패턴으로 해결 가능한 테스트 실패

### 반드시 중단할 중요 문제

- 요구사항을 변경해야 테스트 통과 가능
- API 계약, DB 스키마, 트랜잭션 경계 변경 필요
- 인증·권한 정책 변경 필요
- 동시성 정합성 미보장
- 데이터 손실 또는 중복 처리 가능성
- 테스트 삭제·비활성화·약화 필요
- 성능 목표 미달 또는 운영 장애 가능성
- ADR과 실제 구현 충돌
- 임시 우회와 근본 해결 중 선택 필요
- Issue 범위 밖 또는 보호 파일 변경 필요
- 현재 Issue와 무관한 테스트 실패로 전체 검증 불가

중요 문제를 단순 수정으로 숨기지 않는다.

## 9. Troubleshooting Gate 보고 형식

```markdown
## Troubleshooting Gate

### 발생한 문제

### 재현 조건

### 확인된 사실

### 추정 원인

### 시도한 방법과 결과

### 영향 범위
- 기능:
- 데이터:
- 트랜잭션:
- 테스트:
- 운영:

### 가능한 선택지

### 권장 방향

### 필요한 Human 결정
```

확인되지 않은 원인을 확정 사실처럼 보고하지 않는다.

## 10. 구현 후 검증

다음 순서로 실행한다.

1. 변경과 직접 관련된 가장 작은 테스트
2. 필요한 통합·동시성·성능 테스트
3. 전체 테스트
4. 전체 빌드
5. diff 검사

기본 명령:

```bash
./gradlew test
./gradlew build
git diff --check
git status
git diff --stat
git diff
```

중복 실행은 줄일 수 있지만 핵심 검증은 생략하지 않는다.

최종 보고에는 다음을 구분한다.

- 테스트가 보장하는 것
- 테스트가 보장하지 않는 것
- 실행한 검증과 결과
- 수행하지 못한 검증
- 미검증 항목

## 11. Commit·Push·PR

READY Issue와 필요한 Human Gate를 통과하고 검증이 완료되면 별도 단계별 승인 없이 다음을 수행한다.

1. Issue 범위의 코드와 테스트만 Commit한다.
2. 작업 브랜치를 Push한다.
3. PR을 생성한다.
4. PR 본문에 다음을 기록한다.
   - `Closes #<Issue 번호>`
   - 구현 내용과 주요 변경 파일
   - 테스트와 빌드 결과
   - 테스트가 보장하는 것과 보장하지 않는 것
   - 직접 검증 결과
   - 의미 있는 트러블슈팅
   - 미검증 항목
   - Issue 범위 이탈 여부
   - ChatGPT 집중 리뷰 대상
   - ChatGPT 문서 보완 필요 항목
5. PR 번호를 Human에게 전달한다.

PR 생성 후에는 전체 결과를 대화에 복사하지 않는다.

## 12. PR 생성 전 BLOCKED 보고

Push 또는 PR 생성 전에 중단된 경우 다음만 보고한다.

- 실패한 명령과 핵심 오류
- 현재 브랜치와 작업 트리 상태
- 생성된 Commit 유무
- 확인된 사실과 추정 원인
- 수행하지 못한 검증
- 필요한 Human 결정 또는 환경 조치

PR 생성 이후 Actions 실패는 PR 번호를 전달하고 ChatGPT가 GitHub에서 원격 로그를 확인하도록 한다.

## 13. Merge 금지

Codex의 작업은 PR 생성과 리뷰 반영까지다.

```text
Codex Merge 금지
ChatGPT Merge 금지
자동 Merge 금지
Human만 Merge 가능
```

Human이 승인했다고 말하더라도 Codex가 실제 Merge 명령이나 GitHub Merge 작업을 수행하지 않는다. 최종 Merge는 Human이 직접 실행한다.
